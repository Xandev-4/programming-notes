Mastering Hoisting & Temporal Dead Zone

The real definition of hoisting, var vs let vs const hoisting, TDZ, function declaration hoisting, function expression trap · June 2025

The Misconception About Hoisting

The internet is full of a wrong definition of hoisting: "JavaScript physically moves variable and function declarations to the top of the file before executing." This is not what happens. Nothing is moved. No code is rearranged.

❌ Wrong Definition

"JavaScript lifts/hoists variable and function declarations to the top of the scope before execution."

Implies code is physically rearranged — it isn't

✅ Real Definition

Hoisting is the result of the creation phase of the Execution Context running before the execution phase. Memory is allocated and (for var) initialised to undefined before any line of code runs.

Code stays exactly where you wrote it

● Why It Matters

If you understand the EC creation phase from Lec 8, hoisting is trivial — 5 minutes of understanding. If you memorised the wrong definition, it will confuse you forever.

Hoisting = creation phase in action
Key Rule Hoisting = the creation phase of the Execution Context. During creation phase, memory is set up for variables and functions before the execution phase runs a single line. That's it. Nothing is lifted. Nothing moves.

Variable Hoisting with var

When you declare a variable with var, the creation phase allocates memory for it and initialises it to undefined — before any execution phase code runs. So accessing a var variable before its declaration line does not throw an error — it returns undefined.

The Surprising Code

console.log("Name is:", name);  // line 1 — before declaration
var name = "Tom";               // line 2 — declared after usage
name = "Tom";                   // line 3 — assignment
console.log("Name is:", name);  // line 4 — after assignment

// Output:
// Name is:          ← undefined printed as blank — NOT an error!
// Name is: Tom

EC Trace — Why This Works

// ── GEC: Creation Phase ────────────────────────────────
// JS scans globally: finds "var name"
// Allocates memory for name in Stack
// Initialises name = undefined   ← KEY STEP

// ── GEC: Execution Phase ───────────────────────────────
// Line 1: console.log("Name is:", name)
//   → name is currently undefined (from creation phase)
//   → prints "Name is: " (undefined renders as blank)

// Line 2: var name = "Tom"
//   → This is a declaration — already handled in creation phase
//   → JS skips the "var name" part, moves on

// Line 3: name = "Tom"
//   → Assigns "Tom" to name in memory
//   → name is now "Tom"

// Line 4: console.log("Name is:", name)
//   → name is "Tom"
//   → prints "Name is: Tom"
Warning This behaviour with var is a design flaw. Code that uses a variable before declaring it is hard to read, error-prone, and a debugging nightmare. This is a primary reason why var is avoided in modern JavaScript. Always use let or const.

Hoisting with let

Variables declared with let ARE processed during the creation phase — memory is allocated. But unlike var, they are NOT initialised to undefined. They exist in memory but have no usable value. Trying to access them before their declaration line throws a ReferenceError. 

console.log(name); // ❌ ReferenceError: Cannot access 'name' before initialization
let name = "Tom";
console.log(name); // "Tom"

EC Trace — Why let Errors

// ── GEC: Creation Phase ────────────────────────────────
// JS scans globally: finds "let name"
// Allocates memory for name in Stack
// Does NOT initialise — name is in an uninitialised state
// (This uninitialised zone is the Temporal Dead Zone)

// ── GEC: Execution Phase ───────────────────────────────
// Line 1: console.log(name)
//   → name exists in memory but has NO value (uninitialised)
//   → JS throws ReferenceError: Cannot access 'name' before initialization

// Line 2: let name = "Tom"
//   → name is now initialised with "Tom"
//   → TDZ for name ENDS here

// Line 3: console.log(name)  ← would work fine if we got here
//   → "Tom"
Note The error message is Cannot access 'name' before initialization — not "name is not defined." This is intentional. JS knows name exists (it was found in the creation phase) but refuses to let you use it before it's initialised. This is correct and expected behaviour.

Hoisting with const

const behaves like let for hoisting — memory is allocated during the creation phase, but not initialised. Additionally, const has an extra rule: it must be initialised at the point of declaration. You cannot declare a bare const and assign later. 

// ❌ SyntaxError: Missing initializer in const declaration
const name;
// JS catches this at parse time (before any execution)
// You MUST do: const name = "something";
// ❌ ReferenceError: Cannot access 'name' before initialization
console.log(name); // try to access before declaration
const name = "Tom";

// Even though const must be initialised at declaration,
// you still cannot access it before that line — TDZ applies here too.

var vs let vs const Hoisting — Full Comparison

// var: memory allocated + initialised to undefined in creation phase
console.log(a); // undefined — no error
var a = 10;

// let: memory allocated, NOT initialised — accessing throws ReferenceError
console.log(b); // ReferenceError: Cannot access 'b' before initialization
let b = 20;

// const: memory allocated, NOT initialised — same ReferenceError
console.log(c); // ReferenceError: Cannot access 'c' before initialization
const c = 30;

// const extra rule: must be initialised at declaration
const d;        // SyntaxError: Missing initializer in const declaration

● var Hoisting

Memory allocated + initialised to undefined. Accessible before declaration — returns undefined. Bug-prone. Avoid.

● let Hoisting

Memory allocated, NOT initialised. In TDZ until declaration line. Accessing before → ReferenceError. Use this.

● const Hoisting

Memory allocated, NOT initialised. In TDZ until declaration line. MUST be initialised at declaration. Use this for constants.

Temporal Dead Zone (TDZ)

The Temporal Dead Zone is the area of code — from the start of the block/scope to the line where the variable is initialised — during which that variable exists in memory but cannot be accessed. Trying to access it inside the TDZ throws a ReferenceError.

Formal Definition

The TDZ for a variable starts at the beginning of the enclosing block and ends at the exact line where the variable is assigned a value (initialised).

{
  // ↓ TDZ for "name" STARTS here (beginning of block)
  //
  // ... some code ...
  //
  console.log(name); // ❌ ReferenceError — inside TDZ!
  //
  // ... some code ...
  //
  let name = "tapaScript"; // ← TDZ for "name" ENDS here (initialised)
  //
  // ... more code ...
  //
  console.log(name); // ✅ "tapaScript" — TDZ is over
}

Multiple Variables Have Independent TDZs

{
  // TDZ for "name" starts here
  // TDZ for "address" starts here

  console.log(name);    // ❌ ReferenceError — name still in TDZ
  let name = "Alice";   // ← TDZ for name ENDS

  console.log(name);    // ✅ "Alice"
  console.log(address); // ❌ ReferenceError — address still in TDZ!

  let address = "Bangalore"; // ← TDZ for address ENDS

  console.log(address); // ✅ "Bangalore"
}
Tip Each variable has its own independent TDZ. The TDZ for name ending does not affect the TDZ for address. The TDZ of a variable runs from the block start to where that specific variable is initialised.

TDZ in Practice — Why It Exists

// TDZ prevents this class of bug:
function getUserData() {
  // Without TDZ (if let acted like var):
  console.log(data); // would silently return undefined — hidden bug
  // ... 50 lines of code ...
  let data = fetchData(); // actual declaration is way down here

  // With TDZ:
  // console.log(data) would throw ReferenceError immediately
  // → you find the bug instantly instead of chasing a silent undefined
}

// TDZ is a FEATURE, not a bug. It forces you to declare before use.
Note TDZ applies to let and const only. var does not have a TDZ — it is initialised to undefined during creation phase, so it is always "accessible" (just returns undefined before assignment). This is the core reason to prefer let/const.

Function Declaration Hoisting

Function declarations are fully hoisted. During the creation phase, the entire function body — not just a reference, not undefined — is stored in Heap memory. This means you can call a function declaration before it appears in your file without any error.

Example — Calling Before Declaration

// Call before declaration — works perfectly:
chase();  // ← invoked on line 1

function chase() {
  console.log("Tom chases Jerry");
  cod();    // ← inner call to cod — also declared below
}

function cod() {
  console.log("Tom caught Jerry");
}

// Output:
// "Tom chases Jerry"
// "Tom caught Jerry"

EC Trace — Why Function Declarations Work Early

// ── GEC: Creation Phase ────────────────────────────────
// Scans globally. Finds:
//   function chase → entire body stored in HEAP. Reference in Stack.
//   function cod   → entire body stored in HEAP. Reference in Stack.
// (Both functions fully available in memory before any execution starts)

// ── GEC: Execution Phase ───────────────────────────────
// Line 1: chase()
//   → Is chase in memory? YES (stored in creation phase)
//   → Create FEC for chase. Execute it.
//
//   FEC[chase] Creation Phase:
//     No local variables. No inner declarations.
//   FEC[chase] Execution Phase:
//     console.log("Tom chases Jerry") → prints
//     cod() → Is cod in memory? YES (from GEC creation phase)
//       → Create FEC for cod. Execute it.
//         FEC[cod] Execution Phase:
//           console.log("Tom caught Jerry") → prints
//         FEC[cod] ends, popped.
//   FEC[chase] ends, popped.
//
// GEC done.
Key Insight Function declarations are the ONLY construct in JavaScript where the entire definition is available before execution starts. var gets undefined. let/const get nothing usable. Function declarations get their full body stored in Heap during creation phase.

Function Expression — The Hoisting Trap

This is one of the most common interview gotchas. A function expression (var test = function() {...}) looks like it should hoist the same way as a function declaration — but it doesn't. It hoists as a variable, not as a function.

The Trap Code

// Expected: "I am being tested" — looks like function hoisting
// Actual: TypeError!

test();   // ← called before declaration

var test = function() {
  console.log("I am being tested");
};

// Error: TypeError: test is not a function

EC Trace — Why it Errors

// ── GEC: Creation Phase ────────────────────────────────
// JS scans globally. It sees: var test = function() {...}
// This is a VARIABLE declaration (var test), NOT a function declaration.
// So: memory allocated for "test" as a variable → initialised to undefined
// (The function body is NOT stored in Heap — it's a variable value, assigned later)

// ── GEC: Execution Phase ───────────────────────────────
// Line 1: test()
//   → test is in memory but its value is currently undefined
//   → You're calling undefined()
//   → TypeError: test is not a function
//   → (Same as typing undefined() in the console)

// Line 3: var test = function() {...}
//   → NOW test is assigned the function as a value
//   → But line 1 already crashed!

Visual Contrast

// ✅ FUNCTION DECLARATION — fully hoisted
function greet() {
  console.log("Hello!");
}
// In creation phase: greet → entire body in Heap ✅
// Callable anywhere in the file

// ❌ FUNCTION EXPRESSION — hoisted as variable (undefined)
var greet = function() {
  console.log("Hello!");
};
// In creation phase: greet → undefined ❌
// Calling before assignment → TypeError: greet is not a function

The Same Trap with let/const

// Function expression with let → TDZ applies → ReferenceError
test(); // ❌ ReferenceError: Cannot access 'test' before initialization
let test = function() {
  console.log("test runs");
};

// Function expression with const → same TDZ → ReferenceError
test(); // ❌ ReferenceError: Cannot access 'test' before initialization
const test = function() {
  console.log("test runs");
};
Warning A function expression (var/let/const test = function() {}) is treated entirely as a variable for hoisting purposes. The function body is not stored during creation phase — only a variable slot is created. Don't confuse it with a function declaration.

How to Tell Them Apart Instantly

// FUNCTION DECLARATION: starts with the "function" keyword
function foo() {}         // declaration — fully hoisted

// FUNCTION EXPRESSION: function assigned to a variable
var foo = function() {};  // expression — variable hoisted as undefined
let foo = function() {};  // expression — TDZ until this line
const foo = function() {}; // expression — TDZ until this line

// ARROW FUNCTION EXPRESSION: also an expression, same rules
var foo = () => {};       // expression — variable hoisted as undefined
let foo = () => {};       // expression — TDZ

Hoisting Inside Functions

Hoisting is not only a global phenomenon. Every function has its own Function Execution Context — and every FEC also has a creation phase. Local variables and inner function declarations inside a function are hoisted within that function's scope. 

function outer() {
  console.log(x); // undefined — var x is hoisted within outer's FEC
  var x = 10;
  console.log(x); // 10

  inner();        // ✅ works — inner() declaration is hoisted in outer's FEC

  function inner() {
    console.log("inner runs");
  }
}

outer();
// FEC[outer] Creation Phase:
//   var x → allocated → undefined
//   function inner → entire body stored in Heap (within outer's scope)

// FEC[outer] Execution Phase:
//   console.log(x) → undefined (from creation phase)
//   x = 10         → x assigned 10
//   console.log(x) → 10
//   inner()        → inner is in memory → create FEC[inner] → "inner runs"
Note Hoisting is always scope-local. A var inside a function is hoisted to the top of that function, not to the global scope. This is why var is function-scoped, while let/const are block-scoped — covered in full in the Scope lecture.

Interview-Ready Answers & Quizzes

These are the exact questions asked in JS interviews. Practice answering them using the execution context perspective — not the "JavaScript lifts code up" explanation.

Quiz 1 — What is the output?

console.log(a);
var a = 5;
console.log(a);

Answer: undefined then 5. During creation phase, a is allocated and set to undefined. First log prints undefined. Assignment runs in execution phase. Second log prints 5.

Quiz 2 — What is the output?

console.log(b);
let b = 10;

Answer: ReferenceError: Cannot access 'b' before initialization. let is in its TDZ. Memory was allocated in creation phase but NOT initialised — accessing it throws.

Quiz 3 — What is the output?

foo();
function foo() {
  console.log("foo runs");
}

Answer: "foo runs". Function declaration is fully hoisted — entire body stored in Heap during creation phase. Calling before declaration is fine.

Quiz 4 — What is the output?

bar();
var bar = function() {
  console.log("bar runs");
};

Answer: TypeError: bar is not a function. bar is a var variable — hoisted as undefined in creation phase. Calling undefined() throws a TypeError.

Quiz 5 — What is the output?

console.log(typeof x);
var x = 42;

Answer: "undefined". typeof on a hoisted-but-unassigned var returns the string "undefined". Note: typeof on a completely undeclared variable also returns "undefined" (doesn't throw) — this is a special behaviour of typeof.

Quiz 6 — What is the output?

function test() {
  console.log(msg);
  var msg = "Hello";
  console.log(msg);
}
test();

Answer: undefined then "Hello". Hoisting happens inside the function's FEC creation phase — msg is allocated and initialised to undefined before the function body executes.

Quiz 7 — What is the output?

let x = "global";

function show() {
  console.log(x);  // what does this print?
  let x = "local";
  console.log(x);
}

show();

Answer: ReferenceError: Cannot access 'x' before initialization. Inside show(), a local let x is declared — so the FEC creation phase creates a TDZ for the local x. The first console.log(x) is inside the TDZ of the local x, not the global one. JS does NOT fall back to global when a local declaration is in TDZ.

The Real Hoisting Definition — Final Form

After everything covered in this lecture, here is the precise, complete definition of hoisting you should use in interviews and in your own thinking: 

// HOISTING:
// The process by which JavaScript allocates memory for variable and
// function declarations in the CREATION PHASE of the Execution Context
// (GEC or FEC), BEFORE the execution phase runs any line of code.
//
// Specifics:
//   var declarations     → allocated + initialised to undefined
//   let declarations     → allocated, NOT initialised (TDZ until declaration line)
//   const declarations   → allocated, NOT initialised (TDZ until declaration line)
//                          + MUST be initialised at declaration
//   function DECLARATIONS → entire body stored in Heap (fully usable immediately)
//   function EXPRESSIONS  → treated as variables — same hoisting as var/let/const
//                          → body NOT stored in Heap during creation phase
Tip When someone asks you about hoisting in an interview: draw a GEC box, label Creation Phase and Execution Phase, then explain what happens to each type of declaration. This shows depth of understanding that "JS lifts declarations up" never demonstrates.

Key Takeaways

  • Hoisting is NOT physically moving code. It is the creation phase of the Execution Context running before the execution phase.
  • var hoisting: memory allocated + initialised to undefined → accessible before declaration (returns undefined). Bug-prone. Avoid.
  • let hoisting: memory allocated, NOT initialised → accessing before declaration throws ReferenceError: Cannot access 'x' before initialization.
  • const hoisting: same as let + must be initialised at the point of declaration → bare const x; throws SyntaxError: Missing initializer in const declaration.
  • Temporal Dead Zone (TDZ) = the zone from the start of the block to the line where a let/const variable is initialised. Any access inside TDZ → ReferenceError.
  • Each variable has its own independent TDZ. One variable's TDZ ending does not affect another's.
  • TDZ only applies to let and const. var has no TDZ.
  • Function declarations are fully hoisted — entire body stored in Heap during creation phase. Callable anywhere in the scope.
  • Function expressions (var/let/const fn = function(){}) hoist as variables — body NOT stored. Before declaration: var → undefined, let/const → TDZ.
  • Calling a function expression before its declaration:
    • with var → TypeError: fn is not a function (undefined is not callable)
    • with let/const → ReferenceError (TDZ)
  • Hoisting happens inside every FEC too — not just the GEC. Local var and function declarations inside a function are hoisted within that function's scope.
  • TDZ Quiz 7 trap: if a local let exists in a function, it creates a TDZ for that local — JS does NOT fall back to the outer scope variable while inside the TDZ.
