Mastering JavaScript Execution Context

Lexical environment, GEC, FEC, creation & execution phases, call stack, stack vs heap memory, garbage collection · June 2025

Why Execution Context Matters

Execution context is the most fundamental part of JavaScript. Almost every "advanced" JS concept — hoisting, scope, scope chain, closures, the this keyword, and the event loop — are only confusing because developers don't have a solid grip on execution context first. This lecture fixes that.

● Hoisting

Why can you use a function declaration before it appears in the file? Execution context's creation phase explains this completely.

● Scope & Closures

Why does an inner function "remember" variables from an outer function even after the outer has returned? Execution context + lexical environment.

● this Keyword

What is this and why does it change value? It is created as part of the execution context — understanding EC explains this.

Note Execution context is Module 2's foundation. Every topic that follows — hoisting, scope, closures, this, async — will reference what you learn in this lecture.

Lexical Environment

Before understanding execution context, you must understand lexical environment. The word lexical means "related to the text" — so a lexical environment describes where your code is physically placed in the source file. 

function showName() {
  let name = "Alice";       // name is LEXICALLY INSIDE showName
  console.log(name);
}
// showName is LEXICALLY INSIDE the file (global level)
// name is LEXICALLY INSIDE showName
showName();

JavaScript doesn't just run code as-is. It first has to translate it to machine code (tokenize → parse → AST → machine code). During this process, JS checks whether each piece of code is lexically valid — does it follow the grammar rules of the language and is it placed correctly?

● Lexical Placement

A variable declared inside a function is lexically inside that function. That function is lexically inside the file.

Physical location in code = lexical position

● Grammar Validation

JS verifies that every piece of code follows the language grammar based on where it is lexically placed. Invalid placement = SyntaxError.

Using a variable before declaration can error — depends on where it's lexically placed

● Multiple Environments

A single file can have many lexical environments — each function creates its own. The currently running one is managed by execution context.

Global level + each function = separate LE
Note Lexical environment = where code physically sits. Execution context = the managed environment for code that is currently running. These two concepts are tightly linked.

What is Execution Context?

Execution context (EC) is the environment in which a piece of JavaScript code is evaluated and executed. It is everything that surrounds the currently running code and helps it run — all the variables available, the value of this, and the outer scope reference.

Think of it this way: the word "context" means "a set of circumstances that helps you understand an event." Execution context is the set of circumstances that helps JavaScript understand and run your current code.

● Global EC (GEC)

Created first, always. Manages all code outside of functions. Only one GEC exists per JS program.

Created even for an empty JS file

● Function EC (FEC)

Created each time a function is called/invoked. One new FEC per function call. Manages code inside that function.

5 function calls = up to 5 FECs (may nest)

● Eval EC

Created by the eval() function. Rarely used. Avoid in modern JS.

Not covered — avoid eval() in practice

Global Execution Context (GEC)

The GEC is created the instant JavaScript loads your file — even before a single line of code is executed, even if the file is completely empty. It has two phases: Creation Phase and Execution Phase.

GEC Phase 1 — Creation Phase

During creation phase, JavaScript does NOT execute any code yet. It sets up the environment:

  • Creates the window object (in browsers) — the global object containing all browser APIs (alert, prompt, setTimeout, etc.)
  • Creates the this keyword and sets its value to window (at the global level).
  • Scans for all variable declarations in the global scope → allocates memory → initialises each with undefined.
  • Scans for all function declarations in the global scope → stores the entire function body in memory (not undefined — the whole function).
// Even with an empty JS file, the browser creates:
console.log(window);       // The global Window object — full of browser APIs
console.log(this);         // Also the Window object at global level
console.log(this === window); // true — same memory reference
Tip At the global level in a browser, this and window point to the exact same object in memory — confirmed by this === window → true. Since both are objects (non-primitive), equality checks their memory reference, not value.

GEC Phase 2 — Execution Phase

Now JavaScript goes line by line and actually runs the code:

  • Assigns actual values to the variables that were initialised as undefined in creation phase.
  • When it encounters a function call, it does NOT continue to the next line immediately — it creates a new Function Execution Context (FEC) for that function and goes inside it.
let name = "Tom";       // In creation phase: name = undefined
                        // In execution phase: name = "Tom"

function showName() {   // In creation phase: entire body stored in memory
  console.log(name);
}
// In execution phase: showName is already in memory, nothing new to do.
// (Function not invoked yet — no FEC created.)

// If we add: showName(); → THEN an FEC is created for showName

Function Execution Context (FEC)

A new FEC is created every time a function is invoked (called). If a function is never called, no FEC is ever created for it — only the GEC exists. Each FEC also has two phases: Creation and Execution. 

let name = "Tom";

function showName() {
  console.log(this.name); // "Tom"
}

showName(); // ← THIS LINE creates the FEC for showName

FEC Phase 1 — Creation Phase

  • Scans the function body for any local variable declarations → allocates memory → initialises with undefined.
  • Scans for any inner function declarations → stores their bodies in memory.
  • Sets up the arguments object with the values passed to the function.
  • Determines the value of this for this function call.

FEC Phase 2 — Execution Phase

  • Assigns actual values to local variables.
  • When an inner function is invoked → creates another FEC for that inner function (nested FECs).
  • Execution of lines after the function call is paused until the FEC completes.
let name = "Tom";

function showName() {
  // FEC for showName created when showName() is called
  // Creation phase: no local variables to allocate here
  // Execution phase: executes console.log(this.name)
  console.log(this.name); // "Tom"
  // FEC ends → control returns to GEC execution phase
}

showName(); // creates FEC
console.log("After showName"); // only runs AFTER FEC completes
Critical Rule When a function is called, execution of the calling scope is paused until the called function's FEC fully completes. Only then does the calling scope resume from the next line. This is why console.log("After") only runs after showName() is done.

Complex EC Trace — Full Walkthrough

Let's trace every execution context phase for a real 18-line example — exactly as the instructor did in the lecture. This is the most important exercise for internalising how JS runs your code.

The Code

console.log("Start");          // line 1

let a = 5;                     // line 2

function testMe() {            // line 3
  console.log("testMe runs");  // line 4
  let b = 10;                  // line 5
  let user = {                 // line 6
    name: "Tapas",
    age: 30
  };

  function testAgain() {       // line 7
    console.log("testAgain runs"); // line 8
  }

  testAgain();                 // line 9  ← invokes testAgain

  console.log("testMe ends"); // line 10
}

testMe();                      // line 11 ← invokes testMe

console.log("End");            // line 12

Step-by-Step EC Trace

1 — GEC: Creation Phase (CP)

  • window object created. this = window.
  • Variable a → allocated in memory → initialised to undefined.
  • Function testMe → entire body stored in memory (Heap).

2 — GEC: Execution Phase (EP)

  • Line 1: console.log("Start") → creates FEC for console.log → prints "Start" → FEC ends.
  • Line 2: a = 5 → assigns value to a in memory (was undefined).
  • Line 3: testMe function definition → already handled in CP, nothing to do.
  • Line 11: testMe() → function is invoked → GEC pauses here → FEC for testMe is created.

3 — FEC[testMe]: Creation Phase (CP)

  • Variable b → allocated → undefined.
  • Variable user → allocated → undefined.
  • Function testAgain → entire body stored in memory (Heap).

4 — FEC[testMe]: Execution Phase (EP)

  • Line 4: console.log("testMe runs") → FEC for log → prints → ends.
  • Line 5: b = 10 → assigns value.
  • Line 6: user = { name: "Tapas", age: 30 } → object stored in Heap, reference stored in user.
  • Line 9: testAgain() → invoked → FEC[testMe] pauses here → FEC for testAgain is created.

5 — FEC[testAgain]: Creation Phase (CP)

  • No local variables or inner functions → nothing to do in CP.

6 — FEC[testAgain]: Execution Phase (EP)

  • Line 8: console.log("testAgain runs") → FEC for log → prints → ends.
  • FEC[testAgain] completes → removed from call stack → FEC[testMe] resumes.

7 — FEC[testMe] resumes

  • Line 10: console.log("testMe ends") → FEC for log → prints → ends.
  • FEC[testMe] completes → removed from call stack → GEC resumes.

8 — GEC resumes

  • Line 12: console.log("End") → FEC for log → prints → ends.
  • GEC completes → program ends.
// Final output in order:
// "Start"
// "testMe runs"
// "testAgain runs"
// "testMe ends"
// "End"

Call Stack — How JS Manages Execution Contexts

JavaScript uses the call stack to manage which execution context is currently active. The call stack is a LIFO (Last In, First Out) data structure — the most recently pushed EC is always the one currently running, and is the first to be popped when done.

Stack Operations for the Complex Example

// Visualising the call stack state at each step:

// JS starts → GEC is pushed:
[ GEC ]

// testMe() called → FEC[testMe] pushed:
[ GEC, FEC[testMe] ]

// testAgain() called inside testMe → FEC[testAgain] pushed:
[ GEC, FEC[testMe], FEC[testAgain] ]

// testAgain finishes → FEC[testAgain] popped:
[ GEC, FEC[testMe] ]

// testMe finishes → FEC[testMe] popped:
[ GEC ]

// All global code done → GEC popped:
[]   ← empty stack, program ends

console.log Creates an FEC Too

// Every function call — even built-ins — creates an FEC.
// So each console.log() call temporarily adds an FEC to the stack:

[ GEC ]
→ console.log("Start") called:
[ GEC, FEC[log] ]
→ log completes, popped:
[ GEC ]
→ ... and so on
Note The call stack is directly visible in browser DevTools → Sources tab → "Call Stack" panel. When you pause at a breakpoint, you'll see the exact stack of ECs at that moment. This is an essential debugging skill.

Stack Overflow

// Infinite recursion → stack never empties → overflow:
function infinite() {
  return infinite(); // keeps pushing FECs, never pops
}
infinite();
// RangeError: Maximum call stack size exceeded

Stack vs Heap Memory

JavaScript manages memory in two distinct locations: the Stack and the Heap. Where a value lives depends on whether it is a primitive or a non-primitive (reference type).

● Stack Memory

Stores primitive values (numbers, strings, booleans, undefined, null, Symbol, BigInt) and references (memory addresses) to non-primitive values. Access is fast. Fixed size. LIFO structure.

Variables holding primitives live here

● Heap Memory

Stores non-primitive values — objects, arrays, and function bodies. No fixed size. Dynamically allocated. Accessed via a reference pointer stored in the Stack.

Objects, arrays, function definitions live here

● Reference Pointer

When you create an object, the object's data goes to the Heap at some address (e.g., 0xA1B2). The Stack stores the variable name + that address. The variable points to the Heap.

user → 0xA1B2 → { name: "Tapas" }

Memory Layout for the Complex Example

// Code:
let a = 5;
function testMe() {
  let b = 10;
  let user = { name: "Tapas", age: 30 };
  function testAgain() {
    console.log("testAgain runs");
  }
  testAgain();
}
testMe();
// STACK (simplified):
// ┌──────────────────────────────┐
// │ GEC                          │
// │   a = 5               (primitive → value directly in stack)
// │   testMe → 0xHEAP_01  (reference to heap)
// ├──────────────────────────────┤
// │ FEC[testMe]                  │
// │   b = 10              (primitive → value directly in stack)
// │   user → 0xHEAP_02    (reference to heap)
// │   testAgain → 0xHEAP_03 (reference to heap)
// ├──────────────────────────────┤
// │ FEC[testAgain]               │  ← currently running
// └──────────────────────────────┘

// HEAP:
// 0xHEAP_01 → function testMe() { ... }    ← function body
// 0xHEAP_02 → { name: "Tapas", age: 30 }  ← object data
// 0xHEAP_03 → function testAgain() { ... } ← function body

Why Primitives vs References Behave Differently

// Primitive (Stack) — copying creates an INDEPENDENT copy:
let x = 10;
let y = x;    // y gets its OWN copy of 10 in the stack
y = 99;
console.log(x); // 10 — x is unaffected ✅

// Non-primitive (Heap) — copying copies the REFERENCE (both point to same heap location):
let obj1 = { name: "Alice" };
let obj2 = obj1;   // obj2 gets the SAME heap address as obj1
obj2.name = "Bob";
console.log(obj1.name); // "Bob" — obj1 was affected! Both point to the same object in heap
Note This is exactly pass by value vs pass by reference from Lec 2 — now you understand why it works that way. Primitives copy their stack value. Objects copy their Heap address (reference). Same concept, deeper explanation.

Garbage Collection

When an FEC completes and is popped off the stack, all of its local primitive variables are immediately removed from the stack. But what about objects in the Heap? The Heap doesn't automatically free memory when an FEC ends — that is the job of the Garbage Collector.

JavaScript's garbage collector automatically frees Heap memory that is no longer reachable — i.e., no variable in any active execution context holds a reference to that Heap location anymore. The most common algorithm is Mark-and-Sweep.

● Mark Phase

The GC starts from "roots" (global variables, active stack frames) and marks every object reachable from them.

Walk all references, mark as "alive"

● Sweep Phase

Any Heap object NOT marked is considered unreachable and its memory is reclaimed (freed).

Unmarked = garbage = free that memory

● Automatic

You don't manage this manually in JS (unlike C/C++). The engine handles it. Your job: don't accidentally keep references to objects you no longer need.

Setting obj = null removes the reference
function createUser() {
  let user = { name: "Alice", age: 30 }; // object created in Heap
  return user;  // reference returned to caller
}

let person = createUser();
// createUser's FEC ends, popped from stack
// But user object in Heap is still alive — person holds a reference to it

person = null;  // remove the reference
// Now no variable points to that Heap object → GC will reclaim it
Note Memory leaks happen when objects in the Heap remain referenced (accidentally) after you're done with them. Common sources: global variables holding large objects, event listeners never removed, closures keeping unnecessary references. Covered in depth in the advanced module.

Execution Context Phases — Complete Summary

Here is the complete picture of what happens in each phase, for both GEC and FEC:

Creation Phase (both GEC and FEC)

// In this phase, NO code is executed. Memory is set up.

// FOR GEC:
// 1. window object created (browser) / global object created (Node.js)
// 2. this keyword created → points to window (browser global)
// 3. All var/let/const declarations scanned → memory allocated → initialised to undefined
// 4. All function DECLARATIONS scanned → entire body stored in Heap

// FOR FEC:
// 1. arguments object created (contains values passed to the function)
// 2. All local var/let/const declarations → memory allocated → initialised to undefined
// 3. All inner function DECLARATIONS → entire body stored in Heap
// 4. this keyword value determined (depends on how function was called)

Execution Phase (both GEC and FEC)

// In this phase, code runs line by line.

// 1. Values are assigned to variables (overwriting the initial undefined)
// 2. Expressions are evaluated
// 3. When a function CALL is encountered:
//    a. Current phase PAUSES
//    b. A new FEC is created for the called function
//    c. That FEC runs completely (creation + execution phases)
//    d. FEC is popped from call stack
//    e. Current phase RESUMES from next line
// 4. When the phase ends:
//    - GEC ends → program ends
//    - FEC ends → control returns to calling context
Key Insight The reason function declarations are usable before their definition (hoisting) is because in the creation phase, their entire body was already stored in memory. By the time the execution phase runs and encounters the call, the function is already there.

Hoisting — The Execution Context Explanation

Now that you understand the creation phase, hoisting is simple: during the creation phase, variable declarations and function declarations are processed before any code executes. This makes it look like they were "hoisted" (moved) to the top — but they weren't moved. The creation phase just ran first. 

// Function DECLARATION — callable before definition (fully hoisted in creation phase):
greet(); // ✅ works! "Hello!" — function body already in memory from creation phase

function greet() {
  console.log("Hello!");
}

// Variable declared with var — accessible before assignment (hoisted as undefined):
console.log(x); // undefined (not ReferenceError) — memory allocated in creation phase, value = undefined
var x = 10;
console.log(x); // 10

// Variable declared with let/const — NOT accessible before declaration:
console.log(y); // ❌ ReferenceError: Cannot access 'y' before initialization
let y = 20;
// let/const are in the "Temporal Dead Zone" until the line where they are declared
Note This is only a preview — hoisting gets its own dedicated lecture with full detail. The key takeaway here: hoisting is simply a result of the creation phase running before the execution phase. Nothing is magically moved.

The instructor gave this code as a task. Trace the complete execution context for it — identify all GEC and FEC phases, what goes into memory during creation phase, and the order of execution. Do this on paper before running the code. 

// TASK: Trace ALL execution contexts, creation phases, and execution phases for this code.

console.log("Global start");

let result = 0;

function add(a, b) {
  let sum = a + b;
  return sum;
}

function calculate() {
  let x = 10;
  let y = 20;
  result = add(x, y);   // ← what happens here?
  console.log("Result:", result);
}

calculate();

console.log("Global end");

Answer (expand after trying yourself)

// ── GEC: Creation Phase ────────────────────────────────────────────
// result → undefined (stack)
// add    → function body in Heap, reference in stack
// calculate → function body in Heap, reference in stack

// ── GEC: Execution Phase ───────────────────────────────────────────
// console.log("Global start")   → prints "Global start"
// result = 0                    → result updated from undefined to 0
// [add definition line skipped — already in memory]
// calculate()                   → GEC PAUSES → FEC[calculate] created

  // ── FEC[calculate]: Creation Phase ─────────────────────────────
  // x → undefined
  // y → undefined

  // ── FEC[calculate]: Execution Phase ────────────────────────────
  // x = 10
  // y = 20
  // add(x, y) called → FEC[calculate] PAUSES → FEC[add] created

    // ── FEC[add]: Creation Phase ───────────────────────────────
    // a → undefined (will be 10 from argument)
    // b → undefined (will be 20 from argument)
    // sum → undefined

    // ── FEC[add]: Execution Phase ──────────────────────────────
    // a = 10, b = 20 (from arguments)
    // sum = 10 + 20 = 30
    // return 30 → FEC[add] ends, popped from stack

  // FEC[calculate] resumes: result = 30
  // console.log("Result:", 30) → prints "Result: 30"
  // FEC[calculate] ends, popped from stack

// GEC resumes:
// console.log("Global end") → prints "Global end"
// GEC ends

// Final output:
// "Global start"
// "Result: 30"
// "Global end"

Key Takeaways

  • Lexical environment = where code is physically placed in the source file. JS validates grammar based on lexical placement.
  • Execution context (EC) = the environment managing code that is currently running. Contains variables, this, and outer scope reference.
  • Two main types: GEC (global — always one) and FEC (per function call — many possible).
  • Every EC has two phases: Creation Phase (memory setup, no execution) and Execution Phase (line-by-line code runs).
  • GEC Creation Phase: creates window, creates this = window, allocates all global variables as undefined, stores all global function bodies.
  • FEC is only created when a function is called — never just by defining it.
  • When a function is called, the current phase pauses until the new FEC fully completes.
  • Call stack is LIFO — manages all ECs. Currently running EC is always on top. Popped when it returns.
  • Stack memory: primitives and references (addresses). Fast, fixed, LIFO.
  • Heap memory: objects, arrays, function bodies. Dynamic size. Accessed via reference from stack.
  • Copying a primitive → independent copy (pass by value). Copying an object → copies the reference (pass by reference) — both point to the same Heap location.
  • Garbage collection: JS engine automatically reclaims Heap memory that has no more references from any active EC.
  • Hoisting is explained by the creation phase — function declarations fully stored, var initialised as undefined, let/const in Temporal Dead Zone.
