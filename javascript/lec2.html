Lecture 02

Variables & Data Types in JavaScript

How variables work, naming rules, let/var/const, primitive vs non-primitive types, and memory · June 2025

What are Variables?

A variable is a named storage that holds a value. Think of it as a labelled box — the label is the variable name, and whatever you put inside is the value. When you need that value later, you just refer to the box by its name.

● The Box

The variable itself — a named slot in memory where data lives.

e.g. fruit is the box name

● The Value

What you put inside the box. Can be a number, text, boolean, object, etc.

e.g. "mango" is the value inside fruit

● The Specifier

The keyword that controls who can access the box and whether the value can change.

e.g. let, const, var
// Syntax: specifier variableName = value;
let fruit = "mango";
console.log(fruit); // mango
Note Variables store data so that you can reuse it throughout your program. Without variables, every piece of data would be hardcoded and unmanageable.

Value Assignment

The = sign is the assignment operator — it puts a value into the variable. Left side = variable name. Right side = value being assigned. This is not "equals" in the mathematical sense — it means "store this value in this box."

Basic Assignment

let fruit = "mango";
// fruit → "mango"

Reassignment — Replacing a Value

When you assign a new value to the same variable, the old value is wiped out and replaced.

let fruit = "mango";
console.log(fruit); // "mango"

fruit = "kiwi";     // reassignment — no let/const/var keyword
console.log(fruit); // "kiwi"  ← mango is gone

Assigning One Variable to Another

When you assign one variable to another, what gets copied depends on the type of value (primitive vs non-primitive — covered below).

let vegetable = "carrot";
let fruit = "mango";

fruit = vegetable;          // fruit now gets a COPY of vegetable's value
console.log(fruit);         // "carrot"
console.log(vegetable);     // "carrot" — vegetable is UNCHANGED
Note For primitive values, assignment copies the value. The right-side variable stays completely untouched. This is called pass by value.

Declare Now, Assign Later

You can declare a variable first and assign a value later. Until you assign, the variable holds undefined.

let age;
console.log(age);   // undefined

age = 22;
console.log(age);   // 22

Multiple Declarations at Once

let name, salary, department;
name = "Alice";
salary = 50000;
department = "Engineering";
Tip Rule of thumb: declare a variable only once. You can assign/reassign values as many times as needed, but don't re-declare using a keyword like let or const again for the same name.

Naming Rules & Conventions

JavaScript has strict rules for valid variable names. Break these rules and the engine throws a syntax error. Follow these conventions and your code becomes readable.

Hard Rules (must follow)

  • Name must contain letters, digits, $, or _
  • First character must NOT be a digit
  • No spaces allowed in a name
  • Cannot use reserved keywords: if, for, let, const, class, return, etc.
  • Variable names are case-sensitive — myName and myname are two different variables

Valid vs Invalid Examples

let $price = 99;        // ✅ valid — $ is allowed
let _count = 0;         // ✅ valid — _ is allowed
let myName = "Alice";   // ✅ valid — camelCase

let 2cool = "nope";     // ❌ INVALID — starts with digit
let react-play = "nope";// ❌ INVALID — hyphen not allowed
let for = 5;            // ❌ INVALID — reserved keyword

Conventions (best practices)

  • camelCase — start lowercase, capitalize each subsequent word: homeAddress, employeeSalary, isLoggedIn
  • Human-readable — salary not xyz123
  • Match your use case — if it's a salary, call it salary, not amount or count
// ❌ Bad names
let a = "John";
let x99 = 75000;

// ✅ Good names
let employeeName = "John";
let employeeSalary = 75000;

let, var, and const

These three keywords all declare variables, but they behave very differently. Understanding them is critical — you should almost always use let or const. Avoid var in modern JavaScript.

● var

Old-school. Can be re-declared and reassigned. Function-scoped. Causes bugs. Avoid it.

Legacy code only — don't use in new JS

● let

Modern. Can be reassigned but cannot be re-declared in the same scope. Block-scoped.

Use when the value will change

● const

Modern. Cannot be reassigned or re-declared. Block-scoped. Signals the value is fixed.

Use when the value should NOT change

var — can be re-declared (problematic)

var address = "Bangalore";
console.log(address); // "Bangalore"

var address = "USA";  // ⚠️ re-declaring the same variable — no error with var
console.log(address); // "USA"
// This causes confusion — you have no idea if address was intentionally re-declared

let — cannot be re-declared

let address = "Bangalore";
console.log(address); // "Bangalore"

let address = "USA";  // ❌ SyntaxError: Identifier 'address' has already been declared

// But REASSIGNMENT is fine:
address = "USA";      // ✅ no keyword, just reassign
console.log(address); // "USA"

const — cannot be reassigned at all

const address = "Bangalore";
console.log(address); // "Bangalore"

address = "USA";      // ❌ TypeError: Assignment to constant variable
// Once assigned, a const value is locked forever
Tip Default rule: use const for everything. Switch to let only when you know the value needs to change. Never use var.

Re-declare vs Reassign — Know the Difference

let score = 10;       // DECLARATION — uses a keyword (let/const/var)

score = 20;           // REASSIGNMENT — no keyword, just new value
                      // ✅ allowed with let, ❌ not allowed with const

let score = 30;       // RE-DECLARATION — keyword used again for same name
                      // ❌ never allowed with let or const

Comments in JavaScript

Comments are lines the JavaScript engine skips entirely. They exist only for humans — to explain code, leave notes, or temporarily disable lines.

Single-line comment

// This is a single-line comment
let age = 22; // You can also comment inline like this

Multi-line comment

/*
  This is a multi-line comment.
  The JS engine ignores everything between /* and the closing star-slash.
  Use these for longer explanations or temporarily disabling blocks of code.
*/
let name = "Alice";
Tip Comment your why, not your what. The code already says what it does. Comments should explain why a decision was made.

Primitive Data Types

Primitive types are the basic building blocks of data in JavaScript. There are 7 primitive types. They are immutable (you can't change the value itself, only replace it) and are passed by value.

1. String

Any text. Must be wrapped in single quotes '', double quotes "", or backticks ``.

let name = "Alice";
let greeting = 'Hello!';
let template = `My name is ${name}`; // template literal (ES6)
console.log(template); // "My name is Alice"

2. Number

All numeric values — integers and decimals (floats). JavaScript does NOT have separate int/float types.

let age = 22;
let pi = 3.14;
let negative = -100;
console.log(age + pi); // 25.14

3. Boolean

Only two possible values: true or false. Used for conditions and logical checks.

let isLoggedIn = true;
let hasPermission = false;
console.log(isLoggedIn); // true

4. undefined

A variable that has been declared but never assigned a value. JavaScript sets it to undefined automatically.

let salary;
console.log(salary); // undefined — declared but no value assigned yet

5. null

Intentionally empty. The variable is declared AND assigned — but the value is explicitly "nothing". Unlike undefined, null is assigned on purpose by the developer.

let salary = null;
console.log(salary); // null — declared + assigned, but value is intentionally nothing
Note undefined = "I forgot to give this a value." null = "I deliberately set this to nothing." They look similar but mean different things.

6. BigInt

For very large integers that exceed the safe limit of the Number type. Append n to any integer literal.

let bigNumber = 9007199254740991n;
let another = BigInt("123456789012345678901234567890");
console.log(bigNumber); // 9007199254740991n

7. Symbol

Creates a unique identifier. Every Symbol is guaranteed to be different from every other Symbol — even if they have the same description. Used to create unique keys for objects.

let id1 = Symbol("id");
let id2 = Symbol("id");
console.log(id1 === id2); // false — always unique, even with same description
Note You'll use String, Number, Boolean, undefined, and null constantly. BigInt and Symbol come up in more advanced scenarios — don't stress about them now.

Non-Primitive (Reference) Types

Non-primitive types are complex data structures built using primitive values. They are also called reference types because variables don't store the value directly — they store a reference (memory address) to where the value lives in the heap.

● Object

Key-value pairs. Groups related data together. The most fundamental complex type.

e.g. a student with name, age, isEnrolled

● Array

Ordered list of values. Values accessed by index (0-based). Can hold any type.

e.g. a list of 5 numbers: [1, 2, 3, 4, 5]

● Function

A reusable block of code. In JS, functions are also objects — first-class citizens.

e.g. a function that calculates tax

Object Example

An object is a collection of key: value pairs wrapped in curly braces {}.

let student = {
  name: "Alice",       // key: "name", value: "Alice" (string — primitive)
  age: 22,             // key: "age", value: 22 (number — primitive)
  isEnrolled: true     // key: "isEnrolled", value: true (boolean — primitive)
};

console.log(student.name);       // "Alice"
console.log(student["age"]);     // 22

Array Example

An array is an ordered list inside square brackets []. Values are accessed by their index (starting at 0).

let scores = [95, 87, 76, 100, 60];

console.log(scores[0]);    // 95  (first element)
console.log(scores[4]);    // 60  (last element)
console.log(scores.length); // 5

Complex objects can nest other non-primitives

let employee = {
  name: "Bob",
  salary: 75000,
  skills: ["JavaScript", "Node.js", "React"],  // array inside object
  address: {                                    // object inside object
    city: "Bangalore",
    country: "India"
  }
};

console.log(employee.skills[0]);        // "JavaScript"
console.log(employee.address.city);     // "Bangalore"

Pass by Value vs Pass by Reference

This is one of the most important concepts in JavaScript. How a value behaves when assigned to another variable depends entirely on whether it's primitive or non-primitive.

Pass by Value — Primitives

When you assign a primitive variable to another variable, a copy of the value is made. Changes to one do NOT affect the other.

let a = 10;
let b = a;      // b gets a COPY of a's value

b = 20;         // changing b...

console.log(a); // 10 — a is unchanged
console.log(b); // 20 — only b changed

Pass by Reference — Non-Primitives

When you assign a non-primitive (object/array) to another variable, both variables point to the same place in memory. Changing one WILL affect the other.

let obj1 = { name: "Alice" };
let obj2 = obj1;   // obj2 points to the SAME object in memory, not a copy

obj2.name = "Bob"; // changing obj2...

console.log(obj1.name); // "Bob" — obj1 changed too! 😱
console.log(obj2.name); // "Bob"
Warning This trips up a LOT of beginners. If you want an independent copy of an object, use the spread operator: let obj2 = { ...obj1 }; — we'll cover this in depth later.

● Primitive (by value)

A copy is made. Two separate variables with independent values.

Changing one does NOT affect the other

● Non-Primitive (by reference)

Both variables point to the same memory location.

Changing one DOES affect the other

● Deep Copy (fix)

Use spread {...obj} or JSON.parse(JSON.stringify(obj)) to get a truly independent copy.

Changes are isolated, original is safe

Variables in Memory (Stack vs Heap)

When your program runs, it uses your computer's memory. At a high level, there are two memory regions: the Stack and the Heap.

● Stack

Small, fast memory. Stores primitive values directly. Each variable gets its own slot with the actual value.

let age = 22 → stored directly in stack

● Heap

Large, dynamic memory. Stores non-primitive values (objects, arrays). The variable in the stack holds a pointer to the heap address.

let obj = {...} → stack holds address → heap holds data

● Reference

The memory address stored in the stack that points to the actual object data sitting in the heap.

That's why it's called a "reference type"
// Primitives — value lives directly in stack
let x = 42;
let y = x;  // y gets its own copy in the stack — separate slot

// Non-primitives — reference in stack, data in heap
let obj1 = { city: "Bangalore" };
let obj2 = obj1;
// Both obj1 and obj2 in the stack point to the SAME object in the heap
Note You don't manually manage stack/heap in JS — the engine handles it. But understanding this model is what makes pass-by-value vs pass-by-reference finally click.

How JavaScript Reads Your Code

JavaScript doesn't just execute your code top to bottom blindly. It does two passes: a Creation Phase and an Execution Phase. Understanding this explains some weird JS behaviour like hoisting.

Phase 1 — Creation Phase

Before running any code, JS scans the entire file and sets up memory slots for all declared variables and functions. Variables declared with var are set to undefined. Variables with let/const are acknowledged but not initialised (they go into a "temporal dead zone").

Phase 2 — Execution Phase

Now JS executes the code line by line, assigning actual values.

// What you write:
console.log(name);   // What happens? Let's see...
var name = "Alice";
console.log(name);

// What JS does internally (var gets hoisted to top):
// var name = undefined;   ← Creation Phase sets this up
// console.log(name);      → prints: undefined (not an error!)
// name = "Alice";         ← Execution Phase assigns value
// console.log(name);      → prints: "Alice"
// With let — Temporal Dead Zone (TDZ)
console.log(age);   // ❌ ReferenceError: Cannot access 'age' before initialization
let age = 22;
// let is hoisted but NOT initialised — accessing before declaration = TDZ error
Tip Always declare variables at the top of their scope before using them. Relying on hoisting is a code smell and leads to bugs. let and const protect you from this by throwing errors early.

Key Takeaways

  • A variable is a named storage (box). Syntax: specifier name = value;
  • = is the assignment operator, not "equals". Left = variable, right = value.
  • Reassigning replaces the old value. Primitives — old value gone. Objects — same reference.
  • Name rules: letters/digits/$/_ only, no leading digit, no reserved keywords, case-sensitive.
  • Use camelCase, human-readable, use-case-matching names.
  • var — can re-declare. Avoid in modern JS.
  • let — can reassign, cannot re-declare. Use when value changes.
  • const — cannot reassign or re-declare. Default choice.
  • 7 primitive types: String, Number, Boolean, undefined, null, BigInt, Symbol.
  • Non-primitive types: Object, Array, Function — built from primitives.
  • Primitives → pass by value (independent copy). Non-primitives → pass by reference (shared memory).
  • Stack = primitive values stored directly. Heap = objects stored, referenced by pointer.
  • JS reads code in two phases: Creation (sets up memory) then Execution (runs line by line).
  • var hoists as undefined. let/const enter Temporal Dead Zone — access before declaration = error.
