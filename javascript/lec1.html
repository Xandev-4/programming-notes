Lecture 01

Introduction to JavaScript & Environment Setup

What JS is, its history, how to set up your dev env, and the right way to include scripts · June 2025

What is JavaScript?

JavaScript is a programming language that runs both on the client side (browser) and on the server side (Node.js). It is what makes web pages interactive — button clicks, animations, dynamic content updates, form validations — all JS.

● Client Side

Runs in the browser. Handles user interactions, DOM manipulation, and UI updates.

e.g. Button clicks, animations, live form validation

● Server Side

Runs on a server via Node.js. Handles data management, authentication, APIs.

e.g. REST APIs, user auth, database queries

● Beyond Web

Can be used to build mobile apps, games, desktop apps, and AI-powered products.

e.g. React Native, Electron, TensorFlow.js
Note HTML gives structure, CSS gives style, and JavaScript gives behaviour. All three together make a modern web page.

History of JavaScript

JavaScript was created by Brendan Eich in just 10 days in 1995 at Netscape. The fact that one of the most widely used languages today was written in 10 days is both hilarious and humbling.

Timeline

  • 1995 — Brendan Eich creates JavaScript at Netscape in 10 days.
  • 1996 — Microsoft releases JScript for Internet Explorer.
  • 1997 — ES1 standardised by ECMA.
  • 1999 — ES3 introduces Regex and exception handling. Ajax emerges.
  • 2006 — jQuery simplifies JS DOM manipulation massively.
  • 2009 — ES5 brings JSON, Array.filter, Array.reduce.
  • 2010 — AngularJS gains popularity as the first major frontend framework.
  • 2013 — React.js introduced with component-driven architecture.
  • 2015 — ES6 / ES2015 — the "Modern JavaScript" update. let, const, classes, modules, arrow functions, template literals.
  • 2016 — Vue.js rises as a lightweight React alternative.
  • 2017 — ES8 introduces async/await — async programming gets dramatically simpler.
  • 2020 — Deno introduced as a Node.js alternative.
  • 2024–2025 — JS continues to evolve with React Server Components, AI-powered features, and WebAssembly integrations.
Tip ES6 (2015) is the most important milestone. That's when JS became "modern". Most tutorials and jobs expect you to know ES6+ syntax fluently.

Environment Setup

You need three things to start writing JavaScript:

  • VS Code — Code editor. Download from code.visualstudio.com. Install the Live Server extension to preview HTML files in the browser with hot reload.
  • A Browser — Chrome recommended. Right-click → Inspect → Console tab. This is where you'll see JS output and errors.
  • Node.js — Download from nodejs.org. Needed for running JS on the server side and later for npm packages.

Your First Line of JS (in the browser console)

Open Chrome DevTools (F12 or Right-click → Inspect → Console) and type:

console.log("Hello, JavaScript!");

Hit Enter. You'll see Hello, JavaScript! printed. That's it — you just ran JavaScript without any setup. console.log() prints output to the console. Think of it as JS's version of print().

Tip The browser console is your best friend for quick testing and debugging. Get comfortable with it early.

Including JavaScript in HTML

There are multiple ways to include JS in your HTML. Each has trade-offs. Let's go through all of them.

Method 1 — Inline Script (avoid this)

Writing JS directly inside a <script> tag in your HTML. Works, but violates separation of concerns. HTML should structure, JS should behave — keep them in separate files.

<!-- first.html -->
<body>
  <h1>Welcome to JavaScript</h1>
  <script>
    console.log("Hello from inline script");
  </script>
</body>
Note Inline scripts are not reusable. If you need the same JS on 10 pages, you'd have to copy-paste it 10 times. Use external files instead.

Method 2 — External Script in <head> (has a problem)

Linking an external .js file using <script src="..."> inside the <head> tag. This is cleaner but causes a critical bug if your JS tries to access any HTML element.

<!-- first.html -->
<head>
  <script src="test.js"></script>  <!-- ⚠️ Problem here -->
</head>
<body>
  <div id="someId"></div>
</body>
// test.js
document.getElementById("someId").innerText = "I am some div"; // ❌ FAILS

Why does it fail? The browser parses HTML top to bottom. When it hits the <script> in <head>, it pauses HTML parsing, downloads and executes the JS immediately. At that point, the <div id="someId"> hasn't been parsed yet — it doesn't exist in the DOM. So getElementById returns null, and calling .innerText on null throws an error.

Warning Uncaught TypeError: Cannot set properties of null — This is the classic error when your script runs before the DOM element it targets has loaded.

How the Browser Processes HTML + Script

Understanding this flow is fundamental. The browser does these steps:

  • Download HTML → Parse HTML / Build DOM → Download Script → Execute Script → Continue HTML parsing

When <script> is in <head>, JS executes before the DOM is built. That's the problem.

Method 3 — Script at Bottom of <body> (better)

Moving the <script> tag to the very end of <body> guarantees the entire DOM is built before JS runs.

<body>
  <h1>Welcome to JavaScript</h1>
  <div id="someId"></div>

  <script src="test.js"></script>  <!-- ✅ Placed at the bottom -->
</body>
// test.js
console.log("I am a test script");
document.getElementById("someId").innerText = "I am some div"; // ✅ Works!

This works because by the time the browser reaches the <script> tag, all HTML above it is already parsed and in the DOM. But there's still a performance issue: JS only starts downloading after the entire HTML is parsed. For large JS files, this wastes time.

Note Script at bottom of body = correct but not optimal. You're waiting for full HTML parse before even starting the JS download.

async and defer Attributes

The ideal scenario: download JS in parallel with HTML parsing, not sequentially. Both async and defer enable parallel downloading, but they differ in when the script executes.

async

Downloads the script in parallel with HTML parsing. Executes the script as soon as it's downloaded — interrupting HTML parsing if needed. Order of execution is not guaranteed with multiple scripts.

<head>
  <script src="test.js" async></script>
</head>
  • ✅ JS downloads in parallel — faster overall.
  • ❌ Executes immediately when downloaded — may run before DOM is ready.
  • ❌ Multiple async scripts may execute out of order.
  • ✅ Best for: independent scripts like analytics, ads — scripts that don't interact with the DOM.

defer

Downloads the script in parallel with HTML parsing. But defers execution until HTML parsing is fully complete. Order of execution is preserved with multiple scripts.

<head>
  <script src="test.js" defer></script>
</head>
  • ✅ JS downloads in parallel — faster overall.
  • ✅ Executes only after full HTML parse — DOM is always ready.
  • ✅ Multiple defer scripts execute in order.
  • ✅ Best for: most use cases — any script that interacts with the DOM.

Visual Comparison

● Default (no attribute)

Parsing pauses → script downloads → script executes → parsing resumes. Blocks the page.

Problem: Slow, blocks DOM

● async

Script downloads in parallel. Executes immediately when ready — can interrupt parsing.

Use for: Analytics, third-party scripts

● defer

Script downloads in parallel. Executes only after full parse. Order preserved.

Use for: Almost everything else ✅
Tip Default rule: use defer in <head> for all your scripts. It's the best of both worlds — parallel download and safe DOM-ready execution. 
<!-- ✅ The recommended pattern -->
<head>
  <script src="app.js" defer></script>
</head>
<body>
  <div id="app"></div>
  <!-- No script tag at the bottom needed! -->
</body>

JavaScript on the Server Side (Node.js)

Node.js lets you run JavaScript outside the browser — on your machine or a server. This is how backend JS works (APIs, databases, CLI tools, etc.).

Running a JS file with Node

// hello.js
console.log("Hello from Node.js!");
# In your terminal
node hello.js
# Output: Hello from Node.js!

No browser needed. Node.js is essentially a JS runtime built on Chrome's V8 engine, ripped out of the browser and made to run standalone. Once you're comfortable with JS fundamentals, Node is what powers your backend — REST APIs with Express, database connections, and all the backend stuff.

Note In the browser, JS has access to document, window, etc. In Node.js, those don't exist — but you get access to the file system, network, and OS instead.

Key Takeaways

  • JavaScript runs in the browser (client side) and on the server via Node.js (server side).
  • Created in 1995 in 10 days — ES6 (2015) is the modern JS milestone to know.
  • Use console.log() to print/debug. The browser console is your playground.
  • Inline scripts work but violate separation of concerns — use external .js files.
  • Script in <head> without attributes = blocks parsing and may fail to find DOM elements.
  • Script at bottom of <body> = safe but slow (no parallel download).
  • async = parallel download, executes immediately — use for independent scripts.
  • defer = parallel download, executes after full parse — use this by default.
  • Run JS server-side with node filename.js in the terminal.
