Mastering Scope & Scope Chain in JavaScript

Global scope, function scope, block scope, var vs let/const in each scope, scope chain, variable shadowing, loop scoping — with full comparison table · June 2025

What is Scope?

Scope defines where a variable is accessible in JavaScript. It is determined entirely by where the variable is declared in the source code — not where it is used.

The House Analogy

Imagine a house with multiple rooms. Some things (like Wi-Fi) are available everywhere — every room can access them. Other things (like a private diary on a desk in one bedroom) are only accessible in that specific room. JavaScript variables work exactly like this:

  • A variable declared at the global level = Wi-Fi in the house — accessible everywhere.
  • A variable declared inside a function = the private diary — only accessible inside that room (function).
  • A variable declared inside a block = something locked in a drawer — only accessible inside that block.

● Global Scope

Declared outside any function or block. Accessible anywhere in the entire file — inside functions, blocks, everywhere.

● Function Scope

Declared inside a function. Accessible only within that function. Not accessible outside — regardless of whether declared with var, let, or const.

● Block Scope

Declared inside a block {}. Accessible only within that block — only for let and const. var ignores block scope.

Note There is a 4th scope type: Module Scope. Variables inside an ES6 module are scoped to that module. This will be covered in the JavaScript Modules lecture. For now, focus on the first three.

Global Scope

A variable has global scope when it is declared outside of any function or block. It can be accessed from anywhere in the file — inside functions, inside blocks, at the top level. 

let name = "Tapas"; // declared at global scope

function greeting() {
  console.log("Hello,", name); // ✅ accessible inside function
}

{
  console.log("Block:", name); // ✅ accessible inside a block
}

greeting();
console.log("Global:", name); // ✅ accessible at global level

// Output:
// Block: Tapas      (block runs first — it's inline code)
// Hello, Tapas
// Global: Tapas

var in Global Scope → Adds to window Object

Here's a subtle but important distinction: when you declare a variable with var at the global scope, it becomes a property of the window object (in browsers). When you use let or const, it does NOT get added to window. 

// With var:
var age = 30;
console.log(window.age);  // 30 — var added itself to window object

// With let:
let city = "Bangalore";
console.log(window.city); // undefined — let is NOT on window object

// With const:
const country = "India";
console.log(window.country); // undefined — const is NOT on window object
Why? let and const are block-scoped. Even at the global level, they live in an implicit block — not directly on the window object. var is function-scoped and at the global level it has no enclosing function, so it attaches to window. One more reason to use let/const — they don't pollute the global object.

Function Scope

A variable declared inside a function — with var, let, or const — is accessible only within that function. Trying to access it from outside throws a ReferenceError. This rule applies regardless of which keyword you use. 

function toDo() {
  var task = "Learning 40 Days of JS"; // inside function
  console.log(task); // ✅ accessible here
}

toDo();
console.log(task); // ❌ ReferenceError: task is not defined
// Same result with let:
function toDo() {
  let task = "Learning 40 Days of JS";
  console.log(task); // ✅ works
}

toDo();
console.log(task); // ❌ ReferenceError — same error
Common Misconception Beginners often think var declared inside a function is accessible everywhere because "var is global." This is WRONG. Function scope applies to all three keywords. A variable declared inside a function — with any keyword — is NOT accessible outside that function. The "var is different" rule only applies inside a block (not a function). 
// Visual confirmation of the rule:
function scopeTest() {
  var a = 1;
  let b = 2;
  const c = 3;
  console.log(a, b, c); // ✅ 1 2 3 — all accessible inside
}

scopeTest();
console.log(a); // ❌ ReferenceError: a is not defined
console.log(b); // ❌ ReferenceError: b is not defined
console.log(c); // ❌ ReferenceError: c is not defined
// ALL three are inaccessible outside the function — no exceptions

Block Scope

A block is any pair of curly braces {} — used in if, else, for, while, switch, or even a standalone block. Variables declared with let or const inside a block are only accessible within that block. var ignores block boundaries.

let and const — Respect Block Scope

{
  let count = 10;
  console.log(count); // ✅ 10 — accessible inside the block
}

console.log(count); // ❌ ReferenceError: count is not defined
{
  const score = 99;
  console.log(score); // ✅ 99
}

console.log(score); // ❌ ReferenceError: score is not defined

var — Ignores Block Scope (Dangerous!)

{
  var count = 10;      // declared with var inside a block
  console.log(count); // ✅ 10
}

console.log(count); // ✅ 10 — var LEAKS out of the block!
// This looks like block scope is violated — and it is, for var.
// var is FUNCTION scoped, not block scoped.
Key Rule var = function scoped. It is only contained by a function boundary. Block boundaries (if, for, standalone {}) do NOT contain var. let and const = block scoped. They are contained by any block boundary, including functions. This is the single most important scoping rule to memorise.

Block Scope in if-else and for Loops

if (true) {
  let message = "Inside if block";
  console.log(message); // ✅ "Inside if block"
}
console.log(message); // ❌ ReferenceError — message is block-scoped

// With var — it leaks:
if (true) {
  var flag = true;
}
console.log(flag); // ✅ true — var leaks out of the if block!
for (let i = 0; i < 3; i++) {
  // i is block-scoped to the for loop
}
console.log(i); // ❌ ReferenceError: i is not defined

for (var j = 0; j < 3; j++) {
  // j is function-scoped — leaks out
}
console.log(j); // ✅ 3 — var j leaked out of the for loop

Scope Chain

In real programs you have functions inside functions, blocks inside blocks, and complex nesting. When JavaScript encounters a variable access, it uses the scope chain to resolve it — searching from the innermost scope outward until it finds the variable or reaches the global scope.

The Scope Chain Algorithm

  1. Look in the current (nearest) scope. Found? Use it. Done.
  2. Not found? Move one level outward (follow the invisible lexical arrow).
  3. Repeat until you reach the global scope.
  4. Still not found in global scope? → ReferenceError.
Key Scope chain always travels outward only — from inner to outer. It never searches inward. An outer scope cannot access a variable declared inside an inner scope.

Classic 3-Level Example

let globalVar = "I am a global variable";

function outer() {
  let outerVar = "I am an outer variable";

  function inner() {
    let innerVar = "I am an inner variable";

    // Accessing all three variables from the innermost scope:
    console.log(innerVar);  // Found in inner's own scope ✅
    console.log(outerVar);  // Not in inner → go up → found in outer ✅
    console.log(globalVar); // Not in inner → not in outer → found in global ✅
  }

  inner();
}

outer();
// Output:
// "I am an inner variable"
// "I am an outer variable"
// "I am a global variable"

Scope Chain Travels Outward Only

// Trying to access outer's variable from global scope — fails:
function outer() {
  let outerVar = "I am an outer variable";
  inner();
}

outer();
console.log(outerVar); // ❌ ReferenceError: outerVar is not defined
// Global scope cannot look INTO functions. Chain only goes outward.

Full Scope Chain Trace with 3 count variables

var count = 10; // global

function outer() {
  var count = 20; // outer-local

  function inner() {
    var count = 30; // inner-local

    console.log(count); // 30 — found in inner's own scope immediately
  }

  inner();
  console.log(count); // 20 — inner's count gone (FEC ended); outer's count is nearest
}

outer();
console.log(count); // 10 — outer's count gone; only global count remains

Scope Chain — Removing Inner Declarations

var count = 10; // global

function outer() {
  var count = 20; // outer-local (inner's removed)

  function inner() {
    // count NOT declared here
    console.log(count); // 20 — not found in inner → go up → found in outer
  }

  inner();
  console.log(count); // 20 — found in outer's own scope
}

outer();
console.log(count); // 10 — global
var count = 10; // global (both inner and outer removed)

function outer() {
  // count NOT declared here

  function inner() {
    // count NOT declared here
    console.log(count); // 10 — not in inner, not in outer → found in global
  }

  inner();
  console.log(count); // 10 — not in outer → found in global
}

outer();
console.log(count); // 10 — global
Insight Tracing scope chain step by step — "Is it in THIS scope? No → go up. Is it in THAT scope? No → go up..." — is exactly how JavaScript resolves every variable access at runtime. Practice this mental model on every complex code snippet.

Variable Shadowing

Variable shadowing occurs when a variable in an inner scope has the same name as a variable in an outer scope. The inner variable shadows (hides) the outer one — JavaScript prioritises the nearest scope variable and never looks further up the chain for that name. 

let message = "I am doing great"; // outer (global)

function situation() {
  let message = "I am NOT doing great"; // inner — shadows outer
  console.log(message); // "I am NOT doing great" — uses inner (nearest scope)
}

situation();
console.log(message); // "I am doing great" — inner's message is gone (function scope)
                       // outer message is intact and unaffected

Why Shadowing Works This Way

// Inside situation():
//   JS looks for "message" in situation's own scope → FOUND (inner message)
//   → uses "I am NOT doing great"
//   → outer message is hidden (shadowed) — never looked for

// After situation() returns:
//   JS looks for "message" in global scope → FOUND (outer message = "I am doing great")
//   → inner message is gone (function scope ended)
//   → outer message was NEVER modified — shadowing doesn't mutate the outer variable
Tip Shadowing does NOT change the outer variable's value. The inner variable is a completely independent memory allocation with the same name. They coexist — the inner one just takes priority within its scope. Once the inner scope ends, the outer is fully intact.

Shadowing with Blocks

let score = 100; // outer

{
  let score = 200; // inner block — shadows outer
  console.log(score); // 200 — inner scope wins
}

console.log(score); // 100 — back to outer, inner block ended

Shadowing Interview Trap — TDZ + Shadowing

let x = "global";

function show() {
  console.log(x);  // ❌ ReferenceError — NOT "global"!
  let x = "local";
  console.log(x);  // ✅ "local"
}

show();

// Why? Inside show(), a local "let x" is declared.
// The FEC creation phase allocates memory for x (local) → it's in TDZ.
// console.log(x) hits the TDZ of the LOCAL x — JS does NOT fall back to global.
// Once a local shadow exists in any state (including TDZ), it takes full ownership.

var vs let in Loops — A Critical Difference

The difference between var and let in loops is one of the most common interview questions and a real-world source of bugs. The reason comes directly from their scoping rules.

The Problem with var in Loops

// Using var — all callbacks share the SAME variable:
for (var i = 0; i < 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, 1000);
}
// Expected: 0  1  2 (one per timeout)
// Actual:   3  3  3
//
// Why? var is function-scoped. There is only ONE "i" variable.
// When the setTimeout callbacks fire (after 1 second), the loop has
// already finished. i = 3. All three callbacks read the same i = 3.
// Using let — each iteration gets its OWN block-scoped copy of i:
for (let i = 0; i < 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, 1000);
}
// Output: 0  1  2  ✅
//
// Why? let is block-scoped. Each iteration of the for loop creates
// a NEW block, and each block has its OWN independent copy of i.
// Each setTimeout callback closes over its OWN i.
Tip Always use let in for loops. The block-scoping behaviour means each iteration has its own binding — essential for any async code (setTimeout, Promises, event listeners) inside loops.

Synchronous Loop — var Leaks After Loop

for (var j = 0; j < 5; j++) { /* loop body */ }
console.log(j); // ✅ 5 — var j leaked out of the for loop block

for (let k = 0; k < 5; k++) { /* loop body */ }
console.log(k); // ❌ ReferenceError: k is not defined — let k is block-scoped

Test Your Knowledge — Scope Quizzes

Quiz 1 — What is the output?

let name = "Global";

function greet() {
  let name = "Local";
  console.log(name);
}

greet();
console.log(name);

Answer: "Local" then "Global". Inside greet(), the local name shadows the global one. After greet() returns, the local is gone — global prints.

Quiz 2 — What is the output?

var count = 1;

function outer() {
  var count = 2;
  function inner() {
    console.log(count);
  }
  inner();
}

outer();
console.log(count);

Answer: 2 then 1. Inside inner(), no local count → scope chain goes up to outer() → finds count = 2. After outer(), only global count = 1 remains.

Quiz 3 — What is the output?

{
  var x = 10;
  let y = 20;
}
console.log(x);
console.log(y);

Answer: 10 then ReferenceError: y is not defined. var x leaks out of the block (function-scoped). let y is block-scoped — gone after the block closes.

Quiz 4 — What is the output?

function test() {
  if (true) {
    var a = "inside if with var";
    let b = "inside if with let";
  }
  console.log(a);
  console.log(b);
}
test();

Answer: "inside if with var" then ReferenceError: b is not defined. var a is function-scoped — accessible anywhere inside test(). let b is block-scoped to the if block — gone outside it.

Quiz 5 — What is the output?

var count = 10;

function outer() {
  function inner() {
    console.log(count);
  }
  inner();
}

outer();

Answer: 10. No count in inner() → scope chain up to outer() → no count in outer() → scope chain up to global → count = 10 found.

Quiz 6 — What is the output? (Tricky)

let value = "outer";

function check() {
  console.log(value); // line A
  let value = "inner";
  console.log(value); // line B
}

check();

Answer: Line A throws ReferenceError: Cannot access 'value' before initialization. The local let value is in TDZ when line A runs. Even though "outer" exists globally, the local shadow's TDZ takes ownership — JS never falls back to global for a shadowed name in TDZ.

Key Takeaways

  • Scope = where a variable is accessible. Determined by where it is declared, not where it is used.
  • Global scope — declared outside any function or block. Accessible everywhere.
  • var at global scope → added as property to window object. let/const → NOT added to window.
  • Function scope — declared inside a function. Accessible only within that function. Applies to ALL three keywords (var, let, const).
  • Block scope — declared inside a block {}. Only let and const respect block scope. var leaks out of blocks.
  • Key rule: var = function scoped. let/const = block scoped.
  • In the global scope, all three are accessible everywhere (no difference). In function scope, all three are limited to that function. Block scope is where they differ.
  • Scope chain — when a variable is accessed, JS searches from the nearest (innermost) scope outward until global. If not found in global → ReferenceError.
  • Scope chain only travels outward — outer scopes cannot access inner scope variables.
  • Variable shadowing — inner variable with same name as outer hides the outer within its scope. Outer variable is NOT modified. Its value is intact after the inner scope ends.
  • TDZ + shadowing trap: if a local let/const exists in a function — even before its declaration line — JS does NOT fall back to the outer scope variable. TDZ owns the name.
  • var in loops → only one shared variable across all iterations (classic async bug). let in loops → new block-scoped binding per iteration. Always use let in loops.
