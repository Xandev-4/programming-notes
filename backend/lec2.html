Lecture 02

What is a Backend, How Do They Work & Why Do We Need Them?

Series by Sriniously · Video 3 of playlist · Last updated: May 2026

What is a Backend?

In its traditional definition, a backend is a computer which is listening for HTTP, WebSocket, gRPC, or any other kind of request through an open port (e.g. port 80 or port 443), accessible over the internet so that clients or other frontends can connect to it.

What a Server Does

  • Serves content — static files like images, JavaScript, HTML files
  • Serves data — JSON responses from API endpoints
  • Accepts data — receives and processes what clients send to it
Definition We call it a server because it provides or serves some kind of content — whether static files or dynamic data.

Key Characteristics

  • Listens on an open port that is accessible over the internet
  • Can serve any client — browsers, mobile apps, other servers
  • The goal is to get a holistic, physical view of how backend components actually work behind the scenes

How Backends Work

To understand how a backend works in practice, trace a real request from a browser all the way to the server. Every step in the journey has a distinct component with a distinct role.

● Client

The browser or app initiating the request to a domain name.

e.g. browser hits backend-demo.xyz

● Network Hops

DNS → Firewall → Reverse Proxy → Server Process

Each hop has a specific responsibility

● Server Process

The actual runtime (e.g. Node.js) running on localhost on a port.

e.g. localhost:3001

DNS Resolution

The first stop after the browser sends a request to a domain name is the DNS server. DNS translates a human-readable domain name into an IP address that points to the actual machine.

DNS Record Types

  • A Record — Maps a domain/subdomain directly to an IP address
  • CNAME Record — Maps a domain/subdomain to another domain name (alias)

Example Flow

  • Subdomain backend-demo.xyz has an A record pointing to the public IP of an EC2 instance
  • Browser resolves the domain → gets the IP → sends the request to that IP
  • Different subdomains can point to different IPs or services
Note DNS is a huge topic in itself. The key takeaway here is: domain → IP via A record. That IP is where your server lives.

Firewall & Ports

Before a request reaches the server process, it passes through a firewall. On AWS, this is implemented via Security Groups. The firewall controls which ports are accessible from the internet.

Common Ports

  • Port 22 — SSH access to the instance (for terminal/admin operations)
  • Port 80 — HTTP traffic
  • Port 443 — HTTPS traffic

Why This Matters

  • If ports 80 or 443 are not allowed in the security group, the request is blocked at the firewall and never reaches the server
  • Security groups define exactly what traffic is permitted — everything else is denied by default
  • This is the first line of defence for a backend server on cloud infrastructure

Reverse Proxy (Nginx)

After passing through the firewall, the request reaches a reverse proxy. A reverse proxy is a server that sits in front of other servers and manages redirects and configs from a centralised place — so you don't have to change config on every individual server.

What Nginx Does in This Example

  • Listens on Port 80 (HTTP) and redirects to Port 443 (HTTPS) — handled automatically by Certbot/SSL
  • Reads the server_name field to identify which domain is being requested
  • Forwards (proxies) that request to the actual Node.js server running on localhost:3001

Key Nginx Concepts

  • server_name — Identifies the domain/subdomain this config block applies to
  • proxy_pass — Where to forward the matched request (e.g. localhost:3001)
  • Certbot / sbot — Tool used to automatically assign SSL certificates (handles the 80→443 redirect)
  • Multiple servers — One Nginx instance can route traffic to multiple backend services (e.g. frontend on :3000, backend on :3001)
Why Reverse Proxy? Instead of exposing your Node.js server directly, Nginx sits in front — handling SSL termination, routing, and load balancing from one place.

Process Management

  • PM2 — Used to manage running processes on the server (e.g. pm2 list shows all running processes)
  • In the demo: one PM2 process for the frontend (port 3000), one for the backend (port 3001)
  • PM2 keeps processes alive after server restarts and provides process monitoring

Full Request Flow — End to End

Putting it all together — here is the complete journey of a single HTTP request from browser to backend:

  • 1. Browser — User navigates to backend-demo.xyz
  • 2. DNS Server — Resolves the domain name to the EC2 instance's public IP
  • 3. AWS / Cloud Infrastructure — Request is routed to the correct AWS region and instance
  • 4. Firewall (Security Group) — Checks if port 443 (or 80) is allowed; if not, request is blocked here
  • 5. EC2 Instance — Request arrives at the physical (virtual) machine
  • 6. Nginx (Reverse Proxy) — Matches the domain, redirects HTTP→HTTPS, forwards to localhost:3001
  • 7. Node.js Server — The actual server process handles the request and returns a response
Local Dev When developing locally, you skip DNS, firewall, and Nginx entirely — you just hit localhost:3000/users directly. The server logic is identical.

Why Do We Need Backends?

If you strip down the responsibility of a backend to a single word, it is: data. The need to fetch data, receive data, and persist data — and every action that deals with data.

Real-World Example — Instagram Like

  • User clicks the like button on a post
  • App sends a request to the backend server
  • Server identifies who performed the action (authenticated user)
  • Server persists the like action in the database
  • Server identifies whose post was liked
  • Server triggers a notification to that user
  • Friend receives the push notification on their device
Key Insight All of this — from click to notification — requires a centralised computer that knows the state of all users, all posts, all interactions. That centralised computer is the backend.

Why Centralisation is Necessary

  • Each user's app is customised to their profile and the accounts they follow
  • Notifications must be routed to the correct user — not everyone
  • A centralised server holds global state — data about all users, all actions
  • Without a central server, there is no way to coordinate interactions between users on different devices

How Frontends Work

To understand why backend logic cannot live in the frontend, you first need to understand how frontends actually execute code.

Frontend Request Flow (e.g. Next.js App)

  • Browser navigates to frontend-demo.xyz
  • DNS resolves domain → EC2 IP (same flow as backend)
  • Nginx forwards to localhost:3000 (the Next.js dev/prod server)
  • Browser receives the primary HTML file first
  • Browser then fetches all linked resources: JS files, CSS files, fonts, images — each as a separate request

How the Browser Renders a Frontend

  • CSS files fetched → browser paints the visual styles (backgrounds, fonts, button styles)
  • JS files fetched → browser hydrates the page — attaches all event listeners to buttons and interactive elements
  • Only after hydration do buttons and interactions actually work

The Key Difference from Backends

  • Backend — server receives request, processes it, sends back the result. Processing happens on the server
  • Frontend — server sends the code to the browser, and the browser runs the code. Processing happens on the client's machine
  • The browser is the runtime for frontend code — not a server
Runtime Backend: server is the runtime. Frontend: browser is the runtime. This is the fundamental architectural difference.

Browser Sandbox Environment

  • Browser runtimes are sandboxed — isolated from the OS, processes, and file system
  • Code can only access: the DOM, browser APIs (localStorage, cookies), and external APIs — but only if those APIs have the correct CORS headers
  • This isolation exists because browsers fetch code from remote servers and execute it on the user's machine — without isolation, malicious code could access files and steal data

Why Can't We Write Backend Logic in Frontends?

Even if it were technically possible to run all server logic in the browser, here is why it is a fundamentally bad idea — four concrete reasons:

1. Security Restrictions

  • Browser security policies are extremely restrictive by design
  • Backends often need to access the underlying file system — e.g. writing log files, reading environment variables
  • Browsers do not allow file system access — this is a hard blocker
  • Secret keys, API tokens, and credentials cannot be safely stored in the browser — they would be exposed to the user

2. CORS & External API Restrictions

  • Browsers enforce CORS (Cross-Origin Resource Sharing) — a security policy that restricts JS code from calling APIs on a different domain
  • A frontend app at frontend-demo.xyz can only freely call APIs on the same domain
  • Calling any other domain is blocked by the browser unless that API explicitly allows it via CORS headers
  • Backend servers need to connect to many external services — databases, third-party APIs, payment gateways — without this restriction

3. Database Access

  • Servers use native database drivers (e.g. pg for PostgreSQL, mongodb for MongoDB) which require socket connections, binary data handling, and persistent connections
  • Browsers are not designed to maintain persistent connections to databases
  • Backend servers use connection pooling — a maintained list of reusable DB connections — because creating and destroying a connection per request would overwhelm the database
  • If each browser client connected directly to the DB, the database would be overwhelmed with thousands of simultaneous direct connections — with no pooling or management layer
  • There is no practical way to implement connection pooling or efficient query execution from browser environments

4. Computing Power

  • Frontend apps run on all kinds of devices — smartphones, low-end laptops, devices with 256MB RAM and a single-core CPU
  • Heavy business logic would lag and break on underpowered devices
  • A centralised backend server can be scaled vertically or horizontally as needed — you can add more CPU and RAM to handle load
  • Users don't have to bear the computational cost of the application's logic
Summary Security · CORS · Database Drivers · Computing Power — four reasons why backend logic must live on a server, not in the browser. And that assumes it could even run there in the first place.
