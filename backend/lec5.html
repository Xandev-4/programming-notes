Serialisation & Deserialisation

How data crosses language boundaries — the common format that lets a JavaScript client talk to a Rust server.

The Problem

A browser running a JavaScript app needs to send data to a server running Rust. These two environments have completely different data types — JavaScript is dynamic and loosely typed, Rust is compiled and strictly typed. They cannot directly understand each other's native data structures.

The same problem exists in reverse: the server sends a response, and the JavaScript client needs to parse it into its own format and render UI from it. How does any of this work across completely different machines, languages, and environments?

The Core Problem Two machines in different locations, running different languages, need to send data to each other and make sense of it — in a way that is language-agnostic.

The Solution — A Common Standard

The solution: agree on a common format — a set of rules that both the client and server follow when sending and receiving data. Neither side sends their native data types over the wire. Instead, both sides convert to and from this agreed-upon standard.

● The Flow

  • Client sends data — converts JavaScript object → common format → transmits over HTTP
  • Server receives data — reads common format → converts into its own native type (e.g. a Rust struct)
  • Server sends response — converts Rust struct → common format → transmits over HTTP
  • Client receives response — reads common format → converts back into JavaScript → renders UI
Definition Serialisation — converting data from a native language format into a common transmittable format.
Deserialisation — converting data from a common format back into a native language format.

The OSI Mental Model

When data travels over the internet, it passes through the OSI model layers. At the top is the Application Layer (Layer 7) — where your backend code lives and where JSON is written and read. At the bottom is the Physical Layer (Layer 1) — where data becomes literal 0s and 1s transmitted as voltage signals or light pulses over optical fibre.

Between those two extremes, the data goes through several transformations on the way out: JSON payload → TCP segments → IP packets → data frames → raw bits. On the receiving end, the exact reverse happens — bits reassemble back up through every layer until they arrive at the Application Layer as JSON again.

● What This Means for You

  • Everything between Layer 7 and Layer 1 is handled by the OS, network drivers, and hardware — not your code
  • Your server code writes JSON at Layer 7 and sends it
  • The layers below convert it into packets and bits — you never see or touch that
  • By the time data arrives at the receiving server's application layer — it is already JSON again
  • You only ever deal with: JSON in → process → JSON out
Mental Model Think of it as: Client writes JSON → [network handles everything in between] → Server reads JSON. The format the data takes while physically in transit does not matter to you. It arrives at your code as the common format you agreed upon.

Serialisation Standards

There are many serialisation formats in the industry. They fall into two categories:

Text-Based

Human readable. Easier to debug — you can open the payload in any text editor and read it. Slower to parse and larger in payload size.

JSON, YAML, XML

Binary Format

Not human readable. Significantly faster to parse and smaller in payload size. Used for high-performance systems where speed matters more than debuggability.

Protobuf (Protocol Buffers), Avro

Which to Use?

For HTTP REST API communication between clients and servers, JSON is the standard ~80% of the time. Binary formats like Protobuf are used in gRPC and internal service-to-service communication at scale.

REST → JSON · gRPC → Protobuf

JSON Deep Dive

JSON stands for JavaScript Object Notation. Despite the name, it is not limited to JavaScript — it is used everywhere: HTTP APIs, config files, log files, database storage, and more. Its strongest trait is that it is completely human readable.

● Structure Rules

  • Starts and ends with curly braces { }
  • All keys must be strings — wrapped in double quotes. No exceptions. No numbers, no booleans as keys
  • Key-value pairs separated by a colon :
  • Pairs separated by commas
  • No trailing comma after the last pair

● Value Types

  • String — wrapped in double quotes: "hello"
  • Number — integer or float, no quotes: 42, 3.14
  • Boolean — lowercase: true or false
  • Array — ordered list in square brackets: [1, 2, 3]
  • Object — nested JSON object: { "key": "value" }
  • Null — absence of value: null

● Nested Objects

JSON can be nested to any depth. A value can itself be a full JSON object following the same structure rules. For example, a user object with a nested address field:

  • "name": "Aryan" — string
  • "age": 25 — number
  • "address": { "country": "India", "phone": 345678 } — nested object, same rules apply inside

The nested object follows the exact same rules as the root — curly braces, double-quoted string keys, typed values. There is no special syntax for nesting.

Where JSON is Used HTTP REST API request and response bodies · Configuration files (.eslintrc, package.json) · Application and server log files · Database document storage (MongoDB) · Event payloads in message queues

End-to-End Example — Creating a Book

Consider a simple books API. A client wants to add a new book. Here is exactly what happens at each step, and where serialisation and deserialisation occur.

● Step 1 — Client Serialises and Sends (POST /api/books)

The client has a native JavaScript object representing the new book:

  • { id: 3, title: "Deep Work", author: "Cal Newport" }

Before sending it over HTTP, the client serialises it — converting the JS object into a JSON string that follows all JSON rules: double-quoted keys, typed values, no trailing commas. That JSON string is placed in the HTTP request body and transmitted.

● Step 2 — Server Deserialises and Processes

The server receives the raw JSON string from the request body. It deserialises it — converting the JSON string into its own native data type (a Rust struct, a Python dict, a Node.js object — whatever the server language uses). The server then runs its business logic: validates the data, stores it in the database, appends it to the in-memory list.

● Step 3 — Server Serialises and Responds

The server now has its native list of book objects — e.g. a Rust Vec<Book>. Before sending the response, it serialises the list into a JSON array:

  • The response body is a JSON array — wrapped in square brackets [ ]
  • Each book inside is a JSON object — wrapped in curly braces { }
  • Example: [ { "id": 1, "title": "Clean Code", "author": "Martin" }, { "id": 2, "title": "Deep Work", "author": "Newport" } ]

● Step 4 — Client Deserialises and Renders

The client receives that JSON array string. It deserialises it — converting it back into a native JavaScript array of objects. It then iterates over the array and renders each book in the UI.

Key Takeaway Every time data crosses a language boundary — client to server or server to client — it must be serialised into the common format first, and deserialised back on the other side. JSON is that common format for REST APIs. Neither side cares what language the other is using — they only care that the format is valid JSON.
