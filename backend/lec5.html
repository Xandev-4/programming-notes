Serialisation & Deserialisation

How data crosses language boundaries — the common format that lets a JavaScript client talk to a Rust server.

The Problem

A browser running a JavaScript app needs to send data to a server running Rust. These two environments have completely different data types — JavaScript is dynamic and loosely typed, Rust is compiled and strictly typed. They cannot directly understand each other's native data structures.

The same problem exists in reverse: the server sends a response, and the JavaScript client needs to parse it into its own format and render UI from it. How does any of this work across completely different machines, languages, and environments?

The Core Problem Two machines in different locations, running different languages, need to send data to each other and make sense of it — in a way that is language-agnostic.

The Solution — A Common Standard

The obvious solution: agree on a common format — a set of rules that both the client and server follow when sending and receiving data. Neither side sends their native data types over the wire. Instead, both sides convert to and from this agreed-upon standard.

● The Flow

  • Client sends data — converts JavaScript object → common format → transmits over HTTP
  • Server receives data — reads common format → converts into its own native type (e.g. a Rust struct)
  • Server sends response — converts Rust struct → common format → transmits over HTTP
  • Client receives response — reads common format → converts back into JavaScript → renders UI
Definition Serialisation — converting data from a native language format into a common transmittable format.
Deserialisation — converting data from a common format back into a native language format.

The OSI Mental Model

When data travels over the internet, it passes through the OSI model layers. At the top is the Application Layer — where your backend code lives. At the bottom is the Physical Layer — where data becomes voltage signals transmitted over optical fibre as literal 0s and 1s.

Between those two extremes, the data goes through several transformations — from your JSON payload, to data frames, to IP packets, to raw bits. On the receiving server's side, the exact reverse happens — bits reassemble back up through all the layers until they arrive at the Application Layer as JSON again.

● What This Means for You as a Backend Engineer

You do not need to worry about any of the intermediary steps. That is the responsibility of the network layers below you. Your entire mental model should be:

  • Client writes data in JSON at the Application Layer and sends it
  • Somewhere below, it gets converted into IP packets, data frames, then physical bits — not your concern
  • Those bits travel over the network
  • On the server side, those bits reassemble back through all the layers
  • By the time data reaches your server code at the Application Layer — it is already JSON again
  • You only ever deal with JSON in, JSON out
Mental Model As a backend engineer: Client → JSON → [network magic, not your problem] → JSON → Server. The intermediary format the data takes while in transit does not matter. It arrives as JSON. It leaves as JSON.

Serialisation Standards

There are many serialisation formats in the industry. They fall into two categories:

Text-Based

Human readable. Easier to debug — you can open the payload in any text editor and read it. Slower to parse and larger in payload size.

JSON, YAML, XML

Binary Format

Not human readable. Significantly faster to parse and smaller in payload size. Used for high-performance systems where speed matters more than debuggability.

Protobuf (Protocol Buffers), Avro

Which to Use?

For HTTP REST API communication between clients and servers, JSON is the standard ~80% of the time. Binary formats like Protobuf are used in gRPC and internal service-to-service communication at scale.

REST → JSON · gRPC → Protobuf

JSON Deep Dive

JSON stands for JavaScript Object Notation. Despite the name, it is not limited to JavaScript — it is used everywhere: HTTP APIs, config files, log files, database storage, and more. Its strongest trait is that it is completely human readable.

● Structure Rules

  • Starts and ends with curly braces { }
  • All keys must be strings — wrapped in double quotes. No exceptions. No numbers, no booleans as keys
  • Key-value pairs separated by a colon :
  • Pairs separated by commas
  • No trailing comma after the last pair

● Value Types

  • String — wrapped in double quotes: "hello"
  • Number — integer or float, no quotes: 42, 3.14
  • Boolean — lowercase: true or false
  • Array — ordered list in square brackets: [1, 2, 3]
  • Object — nested JSON object: { "key": "value" }
  • Null — absence of value: null

● Nested Objects Example

JSON can be nested to any depth. A value can itself be a full JSON object following the same structure rules. For example, a user object with a nested address:

  • "name" — string value
  • "age" — number value
  • "address" — a nested object with its own keys: "country": "India", "phone": 345678

The nested object follows the exact same rules as the root object — curly braces, double-quoted string keys, typed values.

Where JSON is Used HTTP REST API request and response bodies · Configuration files (.eslintrc, package.json) · Application and server log files · Database document storage (MongoDB) · Event payloads in message queues

Live Demo — The Full JSON Flow

Here is what the instructor demonstrated using a real API — a POST request to create a book, followed by the server's response with the updated book list. This is exactly what serialisation and deserialisation looks like in a live client-server scenario.

● Request — Client Sends (POST /api/books)

The JavaScript client serialises its object into JSON and sends it in the request body:

  • Opening and closing curly braces wrapping the entire payload
  • Key "id" — number value
  • Key "title" — string value
  • Key "author" — string value
  • All keys are double-quoted strings — exactly as JSON requires

This JSON string travels over the network. The server receives it at the Application Layer — already back in JSON format — and deserialises it into its own native struct (e.g. a Rust struct or a Node.js object) to perform business logic.

● Response — Server Sends Back

After adding the book, the server serialises the updated list back into JSON and sends it as the response body:

  • The response is a JSON array — wrapped in square brackets [ ]
  • Each element in the array is a JSON object — wrapped in curly braces { }
  • Each object has double-quoted string keys with number and string values
  • The client receives this JSON array, deserialises it into a JavaScript array of objects, and renders it in the UI
Key Takeaway Serialisation and deserialisation are the techniques by which data is converted to and from a common format — so that clients and servers written in completely different languages can communicate without caring about each other's internal data types. JSON in → server logic → JSON out. That is the entire mental model.
