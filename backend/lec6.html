Authentication & Authorisation

Who are you? What can you do? — From wax seals to JWTs, the complete picture of modern auth.

Authentication vs Authorisation

These two words are used together so often that people conflate them — but they answer two completely different questions.

  • Authentication — answers who are you? — the process of assigning and verifying an identity. e.g. logging into Instagram proves you are the owner of that account.
  • Authorisation — answers what can you do? — the process of determining your permissions within a context after your identity is confirmed. e.g. a regular Instagram user cannot delete someone else's post, but an admin can.
Order Matters Authentication always comes first. You cannot authorise someone whose identity you haven't verified. Auth → Authz, always.

Historical Context

Understanding how authentication evolved helps explain why every modern mechanism exists and what problem it was designed to solve.

● Pre-Industrial — Implicit Trust

Authentication was based entirely on human contextual trust. A respected village elder would vouch for a person. Deals were sealed with a handshake. This worked within small communities but could not scale — the elder was only trusted locally, not across cities or countries.

● Medieval Period — Seals & Tokens

Society needed a system that scaled beyond personal reputation. The solution was the wax seal — a physically unique pattern stamped onto letters and agreements. Seals were the first widely-adopted authentication tokens, working on the principle of something you possess. If you had the seal, you were authenticated. Their weakness: they could be forged — the first known authentication bypass attacks. This prompted more sophisticated mechanisms like watermarks and encrypted codes.

● Industrial Revolution — Passphrases & Shared Secrets

The telegraph became critical infrastructure, and operators needed to validate messages securely. They used pre-agreed static passphrases — the origin of passwords. The principle shifted from something you possess to something you know. A shared secret only known to both parties — far harder to forge than a physical seal.

● 1961 — Digital Passwords & The Plain Text Incident

MIT's Compatible Time-Sharing System (CTSS) introduced passwords for multi-user systems — the first digital authentication. Passwords were stored in plain text. A researcher accidentally printed the entire password file. This incident directly motivated the philosophy: passwords must never be stored in plain text. It led to the development of hashing algorithms — irreversible, fixed-length representations of passwords.

● 1970s — Asymmetric Cryptography

Whitfield Diffie and Martin Hellman invented the Diffie-Hellman key exchange — enabling two parties to establish a shared secret over an untrusted medium (the internet) without ever meeting. This introduced asymmetric cryptography (public/private key pairs) which became the backbone of all modern authentication protocols. Kerberos also emerged — a ticket-based authentication system using trusted third parties, the direct precursor to token-based auth.

● 1990s — MFA & Biometrics

As the internet grew, username/password systems alone became insufficient against brute-force and dictionary attacks. Multi-Factor Authentication (MFA) emerged — combining multiple independent principles:

  • Something you know — password, PIN
  • Something you have — smart card, OTP generator
  • Something you are — fingerprint, retina scan (biometrics)

● 21st Century — Modern Auth

Cloud computing, mobile devices, and API-based architectures demanded more robust authentication. This era produced OAuth 2.0, JWTs, zero-trust architecture, and passwordless authentication (WebAuthn — using hardware-stored public/private keys). The future candidates are decentralised identity (blockchain-based) and post-quantum cryptography — algorithms designed to remain secure even against quantum computers, which will eventually break today's RSA and public-key algorithms.

The Three Core Building Blocks

Before diving into auth types, understand these three components — they appear in every authentication and authorisation flow.

● Sessions

HTTP is stateless by design — every request is isolated, the server remembers nothing about previous requests. This was fine for the early web (static pages). But as the web became dynamic — e-commerce carts, user logins, personalised feeds — statelessness became a bottleneck. Sessions were invented to give servers memory.

When a user logs in, the server creates a unique session ID, stores it alongside the user's data (role, cart items, etc.) in a persistent store (database or Redis), and sends the session ID to the client as a cookie. On every subsequent request, the client sends that cookie back. The server uses the session ID to look up the user's data from the store.

  • Session creation — user logs in → server creates session ID → stores user data with it in Redis/DB → sends session ID to client as a cookie
  • Subsequent requests — client sends cookie → server extracts session ID → fetches user data from store → authenticates and processes request
  • Expiry — sessions are short-lived; after expiry (e.g. 15 minutes), the user must re-authenticate and a new session is created

Sessions evolved from file-based storage → database-backed storage → distributed in-memory stores like Redis or Memcached, which are faster because they store data in RAM rather than on disk.

● JWTs (JSON Web Tokens)

By the mid-2000s, stateful sessions had scaling problems: storing session data for millions of users was expensive, and synchronising it across distributed servers in different regions introduced latency and consistency issues. JWTs were formalised in 2015 as a stateless alternative.

A JWT is a self-contained token — it carries the user's data and a cryptographic signature, all encoded in Base64. The server does not need to store anything. It simply verifies the token's signature with a secret key on every request.

A JWT has three parts separated by dots:

  • Header — metadata about the token itself; primarily the signing algorithm used (e.g. HS256)
  • Payload (Claims) — the actual user data stored in the token:
    • sub — the user's ID (subject)
    • iat — issued at timestamp
    • exp — expiry timestamp
    • Optional: name, role (admin, editor, viewer), or any custom claim
  • Signature — a cryptographic hash of the header + payload, signed with the server's secret key. If anyone tampers with the token's data, the signature verification fails and the server rejects it

● JWT Advantages

  • Stateless — no server-side storage needed; eliminates DB/Redis lookups on every auth check
  • Scalable — in a microservices architecture, any server that has the shared secret key can independently verify the JWT — no central session store to sync
  • Portable — Base64 encoded, URL-friendly, can be stored in cookies or sent in Authorization headers; small enough to fit anywhere

● JWT Disadvantages

  • No revocation — once issued, a JWT is valid until it expires. If a user's account is compromised, the server cannot invalidate their token without changing the secret key — which would log out all users
  • No forced logout — you cannot kick a specific user out before their token expires

● The Hybrid Approach

To solve JWT's revocation problem, many systems use a hybrid approach: verify the JWT's signature as normal (stateless), but also maintain a blacklist of revoked JWT IDs in Redis. On every request, after verifying the signature, check the blacklist. If the JWT ID is in the blacklist, reject it — even if the signature is valid.

The Trade-off Debate If you're doing a Redis lookup anyway to check the blacklist, you've reintroduced statefulness. At that point, why not use sessions which are considered more secure? There's no universal right answer — the advice: use an auth provider (Auth0, Clerk) in production and skip this headache entirely. Build your own only to learn.

● Cookies

A cookie is a small piece of data stored in the user's browser by the server. The server sets a cookie via a Set-Cookie response header. The browser then automatically includes that cookie in every subsequent request to the same server.

This makes cookies a natural transport mechanism for auth tokens. The auth flow with cookies:

  • User authenticates with username + password
  • Server validates credentials → creates a session ID or JWT → sets it as a cookie in the response: Set-Cookie: token=<value>
  • Browser stores the cookie
  • Every subsequent request automatically includes: Cookie: token=<value>
  • Server reads the cookie, validates the token, identifies and authorises the user

A critical security property: a cookie set by one server is only sent to that server. The browser will not send Server A's cookies to Server B. This is enforced by the browser automatically.

Authentication Types

● Stateful Authentication (Session-Based)

The server is responsible for storing and managing session state. The client holds only a session ID (typically in a cookie) — a meaningless token on its own. All the actual user data lives on the server side in Redis or a DB.

  • User logs in → server creates session ID → stores user data in Redis/DB keyed by session ID → sends session ID as cookie
  • On each request → server reads session ID from cookie → looks up user data in Redis/DB → authenticates the user
  • Advantage: full control — revoke any user's session instantly by deleting it from the store
  • Disadvantage: storage cost at scale; synchronisation overhead in distributed systems

● Stateless Authentication (JWT-Based)

The token itself is the state. The server stores nothing. On each request, the client sends the JWT (in Authorization header or cookie) and the server verifies the signature with its secret key — no DB lookup needed.

  • User logs in → server creates JWT with user ID + role → signs it → sends to client
  • On each request → client sends JWT → server verifies signature → extracts user ID and role from payload → processes request
  • Advantage: no storage cost; highly scalable across distributed systems
  • Disadvantage: cannot revoke tokens; security risk if token is stolen

● API Key Authentication

A simple, long-lived static secret string issued to a client (typically a developer or a service, not an end-user). The client includes this key on every API request, usually in a header like X-API-Key: <key> or as a query parameter.

  • No login flow — the key is pre-issued out-of-band
  • Server looks up the key in its database, identifies the caller, and checks their permissions
  • Use case: server-to-server communication, third-party integrations (e.g. Stripe, SendGrid API keys)
  • Risk: if the key leaks, anyone can use it indefinitely — rotate keys regularly and never commit them to source code

● OAuth 2.0

OAuth 2.0 is a delegated authorisation framework — it lets a user grant a third-party application limited access to their data on another platform, without sharing their password. The classic example: "Sign in with Google."

  • User clicks "Sign in with Google" on your app
  • Your app redirects to Google's auth server
  • User authenticates with Google and grants your app specific permissions (scopes)
  • Google redirects back to your app with an authorisation code
  • Your backend exchanges the code for an access token (server-to-server, secret stays hidden)
  • Your app uses the access token to call Google APIs on the user's behalf
OAuth vs Authentication OAuth 2.0 is technically an authorisation framework, not an authentication protocol. It tells you what a user can access, not who they are. OpenID Connect (OIDC) is a thin identity layer built on top of OAuth 2.0 that adds authentication — it provides an ID token (a JWT) that tells you who the user is.

Hashing & Password Storage

Storing passwords in plain text is one of the worst security mistakes possible. If your database is breached, every user's password is exposed. Hashing makes passwords unreadable at rest.

● What is Hashing?

A hash function takes any input string and produces a fixed-length output (the hash). It has three critical properties:

  • Deterministic — the same input always produces the same hash. "password123" always hashes to the same string
  • Fixed length — regardless of whether the input is 3 characters or 10,000, the output length is always the same (determined by the algorithm)
  • Irreversible — you cannot reverse a hash back to the original string. Given the hash, you cannot find the input

When a user registers, you hash their password and store the hash. When they log in, you hash the submitted password and compare it to the stored hash — never storing or comparing plain text passwords.

● Salting

Hashing alone has a vulnerability: rainbow tables — precomputed databases of common password → hash mappings. If two users have the same password, they'll have the same hash. An attacker who cracks one cracks both.

Salting fixes this. A salt is a unique, random string generated per user and appended to their password before hashing. The salt is stored in plain text alongside the hash.

  • User registers with "password123"
  • Server generates a random salt: "xK9#mQ2"
  • Server hashes: hash("password123" + "xK9#mQ2") → stores hash + salt
  • Even if two users have identical passwords, their salts differ → their hashes differ
  • Rainbow table attacks become computationally infeasible
Algorithm Choice Use bcrypt, Argon2, or scrypt for password hashing — not MD5 or SHA-1 which are fast by design and therefore easy to brute force. Password hashing algorithms are intentionally slow to make brute-force attacks impractical.

Authorisation Models

Once a user's identity is verified, the server needs to determine what they are allowed to do. There are three major models for managing permissions.

RBAC

Role-Based Access Control. Permissions are assigned to roles, and roles are assigned to users. A user's access is determined entirely by their role.

Admin → full access · Editor → read/write · Viewer → read only

ABAC

Attribute-Based Access Control. Access decisions are based on attributes — properties of the user, resource, and environment. More granular and flexible than RBAC.

Allow access if: user.department = "engineering" AND resource.sensitivity = "low" AND time = "business hours"

ReBAC

Relationship-Based Access Control. Access is based on the relationship between a user and a resource. Used by systems like Google Drive and Facebook.

User can edit a doc if they are its owner · User can view a post if they follow the poster

Cookie Security Flags

When the server sets a cookie, it can attach flags that control how the browser handles it. These flags are critical for security.

  • HttpOnly — the cookie cannot be accessed by JavaScript (document.cookie returns nothing for this cookie). Prevents XSS attacks from stealing auth tokens. Always set this on session/token cookies.
  • Secure — the cookie is only sent over HTTPS, never plain HTTP. Prevents the cookie from being intercepted over unencrypted connections.
  • SameSite — controls when the browser sends the cookie on cross-origin requests:
    • Strict — cookie is never sent on cross-site requests (most secure)
    • Lax — cookie is sent on top-level navigations (e.g. clicking a link) but not on embedded requests
    • None — cookie is always sent cross-site (requires Secure flag)
Rule of Thumb For auth cookies, always set HttpOnly + Secure + SameSite=Lax as a baseline. This protects against the three most common cookie-based attacks: XSS, MITM, and CSRF.

Common Auth Attacks

  • Brute Force — systematically trying all possible password combinations. Mitigated by rate limiting and account lockout policies.
  • Dictionary Attack — brute force using a list of commonly used passwords. Mitigated by enforcing strong password requirements and salted hashing.
  • Authentication Bypass — skipping the authentication flow entirely via a vulnerability. e.g. SQL injection in a login form: username = ' OR '1'='1 bypasses the password check entirely.
  • Session Hijacking — stealing a valid session ID (via XSS or network sniffing) and impersonating the user. Mitigated by HttpOnly + Secure cookie flags and HTTPS.
  • CSRF (Cross-Site Request Forgery) — tricking a logged-in user's browser into making an unintended request to the server using their cookies. Mitigated by SameSite cookie flag and CSRF tokens.
  • Timing Attacks — measuring how long a server takes to respond to infer valid usernames or passwords (e.g. a login check that returns faster for invalid usernames than invalid passwords leaks that the username exists). Mitigated by constant-time comparison functions.

Auth Providers — Production Advice

Authentication is extremely easy to get wrong. The list of attack vectors, edge cases, and cryptographic subtleties is enormous. In production systems, the strong recommendation is to delegate to a dedicated auth provider.

  • Auth0 — battle-tested, feature-rich, widely adopted
  • Clerk — modern alternative with better DX for newer stacks
  • Firebase Auth — Google-backed, good for mobile and web apps
  • Supabase Auth — open-source, PostgreSQL-native
When to Roll Your Own Build your own auth only for learning — to understand how sessions, JWTs, hashing, salting, and OAuth flows actually work under the hood. In any real production system of medium or larger complexity, use an auth provider. Their entire job is getting this right so you don't have to.
